# 在具体 Agent + Skill 项目中落地可信等级

## 1. 目标

将通用分析方法应用到一个真实业务，并把最终可信判断逻辑融入现有技术体系，而不是额外建设一套统一运行框架。

## 2. 准备材料

使用置信度设计通用skill 前，尽量准备：

- 业务目标和使用者；
- 输入、输出和关键中间结果；
- Agent 提示词、Skill、流程或代码；
- 业务验收标准；
- 开发测试、集成测试和回归测试问题；
- 用户自测或验收阶段确认的问题；
- 现有 SQL、脚本、API、规则引擎和工作流结构；
- 下游如何使用结果以及错误影响。

缺少材料时，Skill 应将其列为待确认问题，而不是替业务方猜测。

## 3. 运行 Skill 后的核心产物

默认生成一份：

```text
confidence-design.md
```

它应包括：

1. 业务背景与支持范围；
2. 结果单元及关键性；
3. 已知问题与风险映射；
4. 每个结果的高、中、低条件；
5. 整体等级组合原则；
6. 判断逻辑的实现建议；
7. 用户定向审查动作；
8. 未决问题和下一步。

对于审查现有方案，可以额外生成：

```text
confidence-design-review.md
```

## 4. 将判断逻辑融入现有系统

不要默认创建独立“置信度计算脚本”。优先选择最接近业务事实和现有处理流程的位置。

### 4.1 SQL 或数据任务

适合：

- 明细与汇总勾稽；
- 行数和字段完整性；
- 重复记录；
- 时间范围和口径检查；
- 数据源之间的一致性。

可以在现有 SQL 中增加检查列、异常表或任务分支。

### 4.2 现有业务代码

适合：

- 文件解析结果检查；
- 字段映射；
- 公式和指标计算；
- 来源映射；
- 输出生成前后的业务规则。

可以直接在已有处理函数、服务层或工作流节点中增加判断逻辑。

### 4.3 API 或权威业务系统

适合：

- 核对客户、订单、合同、财务等权威信息；
- 获取已有审批状态；
- 判断数据是否最新；
- 验证外部业务约束。

### 4.4 Agent 或工作流

适合：

- 判断输入是否超出支持范围；
- 根据风险选择不同处理路径；
- 对中、低结果生成定向审查清单；
- 对关键风险阻断后续动作。

### 4.5 人工审核节点

适合：

- 无法被稳定自动判断的复杂业务语义；
- 法律、财务、合规等高影响结论；
- 需要业务负责人作最终解释的场景。

## 5. 实施步骤

### 步骤 1：确认结果单元

由业务、开发和测试共同确认：

- 哪些输出可以独立判断；
- 哪些是关键结果；
- 哪些依赖上游结果；
- 错误后果是什么。

### 步骤 2：确认历史问题

把重要测试问题转成：

```text
触发条件
→ 影响结果
→ 严重程度
→ 可识别方式
→ 等级影响
→ 人工核查动作
```

### 步骤 3：确认高、中、低条件

业务负责人必须确认：

- 什么条件足以直接使用；
- 什么情况只需定向检查；
- 什么情况必须阻断；
- 哪些风险可以接受；
- 哪些结果绝不能被平均掩盖。

### 步骤 4：选择实现位置

针对每条条件，优先复用已有：

- SQL；
- 代码；
- 日志；
- 测试；
- API；
- 数据血缘；
- 工作流状态；
- 审批流程。

只有现有系统无法承载时，才考虑新增独立组件。

### 步骤 5：输出用户可理解的标记

每个中、低结果至少说明：

- 结果位置；
- 风险原因；
- 影响程度；
- 建议检查动作。

### 步骤 6：补充回归测试

已经发生过的重要问题应进入对应业务仓库的测试体系。测试形式可以是：

- SQL 测试；
- 单元测试；
- 集成测试；
- 固定样例文件；
- API Mock；
- 工作流测试；
- 人工验收用例。

## 6. 变更管理

以下变化应触发可信设计复审：

- Agent 提示词或 Skill 变化；
- 输入 Schema、文件格式、数据源或规模变化；
- 输出结果新增、删除或改口径；
- 模型、工具、API 或处理流程变化；
- 新增开发测试或用户自测问题；
- 原有判断逻辑被修改或移除；
- 业务风险或人工审核要求变化。

## 7. 首批试点建议

选择两个差异明显的业务：

1. 一个 Excel 或结构化数据处理 Agent；
2. 一个 Word 报告或包含语义判断的 Agent。

评估：

- 是否能清楚拆分结果；
- 是否能将真实测试问题转成等级条件；
- 是否能复用现有代码实现；
- 是否真正减少全文检查；
- 中、低标记是否足够具体；
- 开发人员使用 Skill 的成本是否合理。
