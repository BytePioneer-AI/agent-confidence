# 在具体 Agent + Skill 项目中落地可信等级

## 1. 定位

本指南用于将 `agent-confidence-designer` 的分析结果融入某个真实业务项目。

推荐架构不是固定的“验证器 → 状态 → 聚合器”，而是：

```text
业务背景、测试问题、验收标准
→ 设计可信对象和高中低条件
→ 确定逻辑应落在哪个现有模块
→ 在 SQL、代码、API、前端、工作流或人工流程中实现
→ 在业务输出中展示等级、依据、来源和审查动作
```

## 2. 推荐设计包

```text
confidence-design/
├── confidence-contract.yaml
├── known-risk-patterns.yaml
├── confidence-logic.yaml
└── confidence-review-report.md
```

### `confidence-contract.yaml`

定义：

- Agent 业务目标和支持范围；
- 可信对象和可信含义；
- 结果单元、形态和业务阶段；
- 高、中、低条件；
- 业务影响；
- 集合数量和覆盖要求；
- 依赖关系；
- 审查动作；
- 整体结果的业务规则。

### `known-risk-patterns.yaml`

沉淀：

- 开发测试和用户自测问题；
- 根因；
- 是否可泛化；
- 影响结果；
- 对高中低条件的影响；
- 建议处理方式；
- 回归或验收案例。

### `confidence-logic.yaml`

记录：

- 判断逻辑；
- 高、中、低条件；
- 硬性限制；
- 所需数据和证据；
- 建议实现方式；
- 建议实现位置；
- 是否融合到现有流程；
- 对用户的审查提示。

它不是统一验证器接口。

### `confidence-review-report.md`

用于评审：

- 设计成熟度；
- 关键结果是否定义清楚；
- 历史问题是否纳入；
- 哪些条件仍需业务确认；
- 哪些逻辑建议落入现有代码；
- 哪些结果必须保留人工控制点。

## 3. 第一次接入流程

### 步骤 1：准备材料

至少准备：

- Agent 和 Skill；
- 业务背景；
- 输入、输出和中间产物；
- 已有代码、SQL、API 或前端流程；
- 业务验收标准；
- 开发测试和用户自测问题；
- 初稿与终版、人工修正版或业务验收记录。

### 步骤 2：运行通用 Skill

要求输出：

- 业务背景与范围；
- 结果单元清单；
- 可信含义；
- 业务阶段和结果形态；
- 可信判断维度；
- 硬性条件；
- 高、中、低定义；
- 完整性和覆盖要求；
- 来源追溯要求；
- 技术实现建议；
- 人工审查策略；
- 设计缺口。

### 步骤 3：业务确认

业务负责人需要确认：

- 什么结果可以直接采用；
- 哪些主体、时间和口径适用；
- 关键结果和高影响结果；
- 集合结果必须达到的数量和覆盖；
- 哪些问题不得推测；
- 哪些结果必须由责任部门确认。

### 步骤 4：确定代码落点

优先寻找现有位置：

| 业务判断 | 常见落点 |
|---|---|
| 字段来自直接来源还是回退来源 | 现有解析或转换代码 |
| SQL 汇总和明细是否一致 | 现有 SQL 或数据任务 |
| 规则是否匹配成熟模板 | 前端或后端规则解析模块 |
| 测试用例数量和边界覆盖 | 测试数据生成流程 |
| 历史答案主体和时间是否适用 | 检索与答案选择流程 |
| 权威来源核对 | 现有业务 API 或数据服务 |
| 高影响问题转责任部门 | Agent 工作流或审批流程 |

只有现有结构无法承载时，才考虑新增独立组件。

### 步骤 5：展示给用户

推荐每个中、低结果至少展示：

- 结果位置；
- 等级；
- 判断原因；
- 来源路径；
- 具体审查动作；
- 责任部门；
- 是否建议留空或阻断。

## 4. 三种典型落地方式

### 4.1 融入现有代码

适用于 PBA SO 合同转换：

```text
解析源 Excel
→ 在字段生成时记录来源和回退路径
→ 在写入目标 Excel 时标记高、中、低
→ 在 trace 中保存原因和目标单元格
```

不需要单独的置信度服务。

### 4.2 融入规则生成流程

适用于 ECS UAT：

```text
规则解析
→ 判断是否完整匹配已支持模板
→ 生成单条 clean/dirty 数据
→ 检查数量、分支和边界覆盖
→ 在输出 Excel 中标记等级和缺口
```

高、中、低可以直接由 TypeScript 或现有业务代码生成。

### 4.3 融入检索和答案选择流程

适用于供应商调查问卷：

```text
问题抽取
→ 检索历史问题
→ 保存历史原文和来源
→ 检查主体、时间、权威性和完整性
→ 选择直接写入、标黄复核或留空转部门
```

来源追溯元数据应与答案一起写回或进入审查页。

## 5. 集合型结果的实现建议

对于测试用例、候选答案、客户案例等集合，建议在现有生成流程中保留：

- 实际生成数量；
- 期望数量或数量策略；
- 已覆盖维度；
- 缺失维度；
- 重复或同质结果；
- 每个子结果的来源和原因。

数量只是一个信号，不能替代维度覆盖。

## 6. 来源追溯建议

建议统一保留业务可读的来源信息，但不要求统一技术接口。

可选字段：

```text
source_document
source_sheet_or_section
source_record_or_question
source_value_or_answer
legal_entity
source_date
owner_department
rule_or_mapping
transformation_note
target_location
```

实际字段名可由业务自行决定。

## 7. 加权规则的使用

部分业务可以用加权规则分辅助映射高中低，例如供应商问卷：

```text
语义匹配
+ 主体一致性
+ 来源权威性
+ 时效性
+ 多来源一致性
+ 答案完整性
```

但应遵循：

1. 规则分不直接称为概率；
2. 主体错误、无权威来源等硬性问题优先；
3. 高影响内容可以独立要求部门确认；
4. 后续使用初稿与终版结果调整规则。

## 8. 新问题进入体系

开发测试、用户自测或终版对比发现问题后：

1. 复现问题；
2. 定位结果单元；
3. 分类根因；
4. 判断是否可泛化；
5. 明确是否需要业务确认；
6. 修改高中低条件或硬性规则；
7. 决定修改现有代码、规则、API 调用或人工流程；
8. 增加回归或业务验收案例；
9. 更新设计版本。

不要仅把问题追加成列表。

## 9. 变更影响分析

以下变化应复核可信设计：

- 新增或修改 Skill；
- 规则模板变化；
- 输入或输出 Schema 变化；
- 新增结果单元；
- 新增法人主体、客户或业务范围；
- 权威数据源变化；
- 测试用例数量或覆盖要求变化；
- 新的终版修改模式出现。

## 10. 设计检查命令

```bash
python skills/agent-confidence-designer/scripts/validate_confidence_package.py \
  confidence-design --strict

python skills/agent-confidence-designer/scripts/check_issue_coverage.py \
  confidence-design
```

它们只检查设计文件：

- 是否包含可信含义；
- 是否定义高中低；
- 集合型结果是否定义覆盖要求；
- 风险是否完成归因和泛化；
- 判断逻辑是否说明实现位置；
- 是否误写了统一运行状态或统一验证器要求。

## 11. 上线前检查

- 每个关键结果的可信含义已经业务确认；
- 高、中、低均有明确条件；
- 集合型结果有数量和覆盖规则；
- 历史问题已区分可泛化与个案；
- 来源追溯字段满足业务审查需要；
- 高影响结果有责任部门策略；
- 判断逻辑已经确定现有系统落点；
- 没有为了通用化额外强制统一状态、接口或服务；
- 真实样例经过业务复核。
