## 1. 适用范围

本最佳实践用于指导开发人员为业务 Agent + Skill 设计可信等级机制，适用于数据处理、Excel、Word、报表生成、文档抽取、业务分析等场景。

它解决的是：

- 如何把一个交付物拆成可独立验证的子结果；
- 如何把开发测试和用户自测中遇到的问题沉淀成检查规则；
- 如何定义高、中、低可信等级；
- 如何确定哪些内容可以自动放行，哪些内容需要定向人工检查；
- 如何避免由 Agent 自评产生虚假的可信感。

它不解决：

- 用一个统一算法判断所有业务结果是否正确；
- 在缺少业务验收标准时自动定义“正确”；
- 用大模型主观判断替代业务规则、复算和勾稽；
- 证明结果绝对无误。

## 2. 核心定义

### 2.1 可信等级

可信等级表示：

> 对某个结果单元，约定的必要验证是否已充分执行，以及当前证据是否支持该结果按既定业务动作被使用。

本实践使用“高 / 中 / 低”，而不是对用户展示百分比。

可信等级不是模型的主观自信程度，也不是结果语言是否流畅。

### 2.2 结果单元

结果单元是可以独立验收、定位和采取审查动作的最小业务结果，例如：

- 一个字段或一条记录；
- 一个汇总指标；
- 一个 Excel 公式或图表；
- Word 中的一项事实陈述；
- 一段原因分析；
- 一个业务决策；
- 一个文件级结构要求。

拆分粒度应满足：出现问题时能够定位责任、验证方式和人工核查动作。

### 2.3 关键性

关键性描述结果错误后可能造成的业务影响。建议至少使用：

- `critical`：错误可能导致付款、合规、法律、对外披露或重大决策风险；
- `high`：影响核心业务指标或主要结论；
- `medium`：影响一般分析或内部使用，但可通过后续流程发现；
- `low`：格式、非关键描述等低影响内容。

关键性与可信等级是两个维度。一个结果即使当前证据较充分，若业务影响极高，也可以保留人工控制点。

### 2.4 验证状态

所有验证器应统一返回以下状态之一：

| 状态 | 含义 |
|---|---|
| `PASS` | 验证已经执行，检查条件满足 |
| `FAIL` | 验证已经执行，发现明确问题 |
| `NOT_RUN` | 按契约应该执行，但本次没有执行 |
| `UNSUPPORTED` | 当前验证器无法处理该输入或结果 |
| `ERROR` | 验证器自身运行失败 |
| `NOT_APPLICABLE` | 经规则判断，本次检查不适用 |

`NOT_RUN`、`UNSUPPORTED` 和 `ERROR` 都不能当作 `PASS`。

### 2.5 已知风险模式

已知风险模式来自开发测试、集成测试、回归测试、用户自测或试运行中已经确认的问题。它至少包括：

- 问题表现；
- 触发条件；
- 影响的结果单元；
- 严重度；
- 检测方式；
- 命中后的等级影响；
- 对应回归测试。

## 3. 总体原则

### 原则一：历史问题决定检查项，当前检查结果决定等级

不能仅因为某类任务历史上容易出错，就直接把当前结果判低。应先把历史问题转化为当前任务可执行的风险检测或验证规则，再依据本次检查结果判定等级。

### 原则二：高可信必须有正向证据

“没有发现错误”不足以判高。只有必要验证已经执行并通过，才能进入高可信候选。

### 原则三：固定脚本与业务规则优先

可复算、可比对、可校验的内容，应由确定性脚本完成。大模型只补充脚本难以覆盖的语义风险。

### 原则四：未验证不等于正确

缺少验证能力、验证器失败、输入超出支持范围，都应显式暴露。不能因为系统没有能力发现错误，就将结果标记为高。

### 原则五：关键失败不能被平均

关键金额、日期、主体、核心指标、合规结论等任一失败，不得被大量非关键通过项平均掉。

### 原则六：等级必须可解释、可追溯

每个等级都应能够回答：

- 检查了什么；
- 哪些检查通过或失败；
- 证据在哪里；
- 哪些内容没有验证；
- 用户应检查什么。

### 原则七：置信设计与业务版本同步

输入结构、输出字段、业务规则、数据源、工具或模型发生变化时，必须进行可信设计影响分析。

## 4. 职责与架构边界

### 4.1 通用最佳实践

负责统一：

- 术语；
- 设计步骤；
- 高、中、低语义；
- 风险和验证器分类；
- 输出格式；
- 设计审查规则。

### 4.2 通用设计 Skill

负责辅助开发人员：

- 读取业务背景、Agent/Skill 实现、测试记录和验收标准；
- 拆分结果单元；
- 提取已知风险；
- 设计验证器；
- 生成可信等级契约；
- 检查规则覆盖与自洽性；
- 生成业务脚本和测试骨架。

通用 Skill 不直接承担运行时业务校验。

### 4.3 业务固定校验脚本

负责在每次运行后检查具体结果，例如：

- Excel 明细与汇总勾稽；
- 字段映射；
- 日期和金额解析；
- Word 数字与底层表格一致性；
- 必要章节完整性；
- 业务不变量；
- 数据源追溯。

具体脚本由业务开发团队实现和测试。

### 4.4 等级聚合逻辑

聚合规则可以采用统一结构，但其阈值、关键项和降级条件必须由业务契约配置。

## 5. 设计流程

### 第一步：明确业务上下文

至少记录：

- 用户和使用目的；
- 输入、输出和中间产物；
- Agent + Skill 的处理步骤；
- 下游如何使用结果；
- 错误可能造成的影响；
- 目前已测试的输入范围；
- 已知不支持的情况。

### 第二步：建立结果清单

将交付物拆成结果单元，并记录：

- 唯一 ID；
- 描述和位置；
- 关键性；
- 验收标准；
- 上游依赖；
- 是否允许自动放行。

示例：

| 结果单元 | 关键性 | 依赖 | 验收标准 |
|---|---|---|---|
| 清洗后的销售明细 | high | 原始文件 | 记录不遗漏、字段正确、主键唯一 |
| 月度销售总额 | critical | 清洗明细 | 独立复算与汇总一致 |
| 报告中的同比增长率 | high | 月度指标 | 与 Excel 指标一致且期间匹配 |
| 增长原因分析 | medium | 客户、订单、销售数据 | 结论有证据，不能把相关性写成因果性 |

### 第三步：建立依赖关系

记录结果之间的依赖。例如报告结论依赖指标，指标依赖清洗明细。上游结果降级时，下游结果不能忽略该风险。

### 第四步：汇总测试问题

收集：

- 开发单测、集成测试和回归测试问题；
- 用户自测和验收问题；
- 边界场景、异常输入和工具失败；
- 已修复但可能回归的问题。

### 第五步：将问题规则化

对每个问题回答：

1. 触发条件是什么？
2. 影响哪些结果？
3. 能否确定性检测？
4. 应检查触发条件还是最终错误？
5. 命中后判中还是低？
6. 无法检测时是否需要人工控制点？
7. 对应回归测试是什么？

### 第六步：设计验证器

优先顺序：

1. 业务不变量和独立复算；
2. 数据完整性、Schema 和来源追溯；
3. 跨文件或跨系统一致性；
4. 已知缺陷模式检测；
5. 语义辅助检查；
6. 人工门禁。

### 第七步：定义高、中、低规则

对每个结果单元定义：

- 必须运行的验证器；
- 可选验证器；
- 哪些失败直接判低；
- 哪些未执行最多只能判中；
- 哪些条件下允许判高；
- 输入超出支持范围时如何处理。

### 第八步：检查覆盖度

至少检查：

- 每个关键结果是否有必要验证；
- 每个重要历史问题是否有检测器或人工控制点；
- 每个历史问题是否有回归测试；
- 是否区分失败和未验证；
- 是否存在只靠模型自评就能判高的路径。

### 第九步：实现业务脚本并验证

验证器本身也必须测试，包括：

- 正常样本；
- 明确错误样本；
- 边界样本；
- 不支持样本；
- 验证器异常样本。

### 第十步：发布前评审

由业务、开发和测试共同确认：

- 结果拆分和关键性；
- 高可信的充分条件；
- 中、低等级对应的用户动作；
- 业务脚本已实现并通过回归；
- 风险位置能够落到具体字段、单元格、段落或步骤。

## 6. 如何沉淀开发测试与用户自测问题

### 6.1 问题模板

每个问题建议记录：

```yaml
id: RISK-001
source: development_test
issue_reference: BUG-173
summary: 混合日期格式导致月份归类错误
trigger_conditions:
  - 同一日期列同时包含 Excel 日期、文本日期和区域化日期
affected_result_units:
  - normalized_sales_detail
  - monthly_sales_total
severity: critical
detectability: deterministic
validator_id: validate-date-normalization
on_detected: low
on_unverified: medium
regression_tests:
  - tests/test_mixed_date_formats.py
```

### 6.2 可进入可信等级设计的问题

适合纳入的问题通常满足至少一项：

- 有明确触发条件；
- 有可观察的错误表现；
- 能定位到结果单元；
- 能复算、比对或使用规则检测；
- 能明确区分通过、失败和无法验证；
- 与业务错误存在稳定关联。

模糊反馈必须继续拆解。例如“结论不太准确”不能直接成为规则，应进一步确认是数字引用错误、缺少证据、遗漏限制条件还是口径误解。

### 6.3 检测触发条件与检测最终错误

检测到风险触发条件不一定表示结果已经错误。

例如输入存在多种日期格式：

- 若标准化后全部可解析，且月份复算一致，可以通过；
- 若存在无法解析且影响汇总，应判低；
- 若无法确认某些日期的业务含义，应判中并要求定向检查。

设计时应尽量检测最终业务错误，并把触发条件作为辅助证据。

## 7. 验证器设计

### 7.1 确定性验证器

适用于可由固定代码明确判断的内容：

- 文件是否完整；
- 必填字段是否存在；
- 类型、格式和取值范围；
- 记录数、唯一性和守恒关系；
- 公式复算；
- 明细与汇总勾稽；
- 报告数字与来源一致；
- 文件是否可打开；
- 已知缺陷模式。

确定性验证器应成为高可信的主要证据。

### 7.2 语义辅助验证器

适用于：

- 结论是否被提供的数据支持；
- 是否把相关性描述成因果关系；
- 摘要是否曲解原文；
- 文档内部是否有语义矛盾；
- 是否遗漏用户明确要求的内容。

语义验证器必须输出结构化信号、风险位置、原因和依据。其“未发现问题”通常不能单独支持高可信。

### 7.3 人工门禁

以下情况可以设置人工门禁：

- 关键语义结论没有稳定自动验证方法；
- 法律、合规或付款流程要求人工批准；
- 输入超出测试范围但业务允许继续；
- 自动检查存在不可接受的漏报风险。

### 7.4 验证器接口

建议统一输出：

```json
{
  "validator_id": "detail-total-reconciliation",
  "target": "销售汇总表!D42",
  "status": "FAIL",
  "criticality": "critical",
  "risk_type": "reconciliation_mismatch",
  "expected": 8342100,
  "actual": 8312100,
  "evidence": ["sales_detail.csv", "销售汇总表!D42"],
  "message": "汇总金额与明细独立复算相差 30000",
  "review_action": "核对过滤条件和公式引用范围"
}
```

### 7.5 验证器本身的可靠性

验证器需要具备：

- 明确输入和输出；
- 版本控制；
- 单元测试和回归测试；
- 错误处理；
- 不支持场景的显式状态；
- 可重复执行；
- 证据留存。

## 8. 高、中、低可信等级

### 8.1 高

建议定义为：

> 所有必要验证已经完成并通过；没有触发关键已知风险；结果证据可追溯；输入处于明确支持和测试范围内。

高可信应同时满足：

- 关键验证器全部执行；
- 关键验证器全部通过；
- 必要非关键验证器没有失败；
- 没有关键验证器处于 `NOT_RUN`、`UNSUPPORTED` 或 `ERROR`；
- 没有触发严重已知风险；
- 依赖的上游结果满足最低等级要求；
- 结果来源和计算链可追溯；
- 输入在支持范围内。

### 8.2 中

建议定义为：

> 没有发现明确关键错误，但证据不足以自动放行，需要针对系统标注的区域进行人工检查。

典型条件：

- 必要验证器未执行、暂不支持或运行异常；
- 输入处于边界场景；
- 触发一般风险模式，但未检测到最终错误；
- 语义结果缺少充分确定性验证；
- 非关键验证失败；
- 上游结果为中；
- 来源链部分缺失；
- 测试覆盖尚不充分。

### 8.3 低

建议定义为：

> 已发现关键错误、关键依据缺失或严重不受支持的情况，必须人工确认，必要时阻断交付。

典型条件：

- 任一关键验证器失败；
- 明细与汇总不一致；
- 关键输入缺失；
- 关键结果无法追溯；
- 输出之间存在明确冲突；
- 触发严重已知缺陷模式；
- 跳过关键处理步骤；
- 输入明显超出支持范围；
- 关键业务不变量被破坏。

### 8.4 内部原因状态

对外可以只展示高、中、低，但内部应保留原因，例如：

- `confirmed_failure`：已确认失败；
- `insufficient_evidence`：证据不足；
- `validator_unavailable`：验证能力不可用；
- `out_of_scope`：超出支持范围；
- `known_risk_triggered`：命中已知风险；
- `upstream_downgrade`：上游依赖导致降级。

这样可以区分“发现错误”和“没有能力验证”。

## 9. 子结果与整体等级聚合

### 9.1 关键结果优先

推荐基本规则：

```text
任一 critical 结果为低       -> 整体低
任一关键结果为中             -> 整体最多中
任一必要验证未完成           -> 整体不能为高
全部关键结果为高且覆盖完整   -> 才可考虑整体高
```

### 9.2 禁止简单平均

不能把高、中、低映射成数字后直接平均。例如 99 个低影响格式项通过，不能抵消一个付款金额失败。

### 9.3 依赖传播

下游结果等级不能高于关键上游依赖允许的最高等级。例如底层指标为中，引用该指标的事实陈述通常不能为高；底层指标为低，相关结论应同步降级或阻断。

### 9.4 业务动作优先

整体等级应服务于业务动作：

- 是否可以自动进入下游；
- 是否需要抽查；
- 是否必须人工审批；
- 是否应阻断。

## 10. 大模型的适用边界

### 10.1 设计期

大模型适合：

- 阅读业务材料；
- 提取结果单元；
- 归类历史问题；
- 提议验证器；
- 检查规则遗漏；
- 生成配置和脚本骨架；
- 生成回归测试建议。

### 10.2 运行期

大模型可以作为语义风险检测器，但必须：

- 使用固定任务和输出结构；
- 指明风险位置；
- 给出依据；
- 不直接决定最终等级；
- 不以自评概率作为主要证据。

### 10.3 非对称使用原则

- 大模型发现明确风险，可以触发降级或人工检查；
- 大模型“未发现问题”只能作为弱证据；
- 大模型不能单独把关键结果提升为高；
- 可确定性验证的内容不得只交给大模型判断。

## 11. 输出与可解释性要求

每个结果单元建议输出：

```yaml
result_unit: monthly_sales_total
level: low
reason_category: confirmed_failure
location: 销售汇总表!D42
checks:
  - validator: detail-total-reconciliation
    status: FAIL
risk: 汇总金额与明细独立复算不一致
evidence:
  - sales_detail.csv
  - 销售汇总表!D42
review_action: 核对过滤条件、隐藏行和公式引用范围
```

任务级输出应包含：

- 整体等级；
- 高、中、低结果数量；
- 必须检查的具体位置；
- 未执行或不支持的验证；
- 主要风险类型；
- 最小人工核查清单。

## 12. 变更与版本治理

以下变更需要重新审查可信设计：

- Agent 或 Skill 提示词、流程和工具变化；
- 输入或输出 Schema 变化；
- 数据源、计算口径或业务规则变化；
- 新增结果单元；
- 验证器实现变化；
- 新测试问题出现；
- 支持范围扩大；
- 模型或解析组件升级导致行为变化。

可信等级契约、风险模式、验证器和测试应分别版本化，并能追溯到 Agent/Skill 版本。

发布门禁建议包括：

- 新关键结果必须有必要验证；
- 新高/严重风险必须有检测器或人工门禁；
- 已知问题必须有回归测试；
- 高可信路径不得依赖未实现的验证器；
- 变更影响分析已完成。

## 13. 反模式

### 反模式一：让执行 Agent 给自己打分

问题：同一模型既执行又评审，无法识别自身未知错误，结果不可重复和不可审计。

### 反模式二：把“运行成功”当作“结果正确”

文件可生成、脚本无异常，只能证明流程执行完成，不能证明业务结果正确。

### 反模式三：用一个综合百分比掩盖证据不足

没有真实校准样本时，百分比产生虚假精确感，且难以指导用户动作。

### 反模式四：所有检查项做平均

会掩盖关键结果失败。

### 反模式五：未执行检查默认通过

这是最危险的高可信来源之一。

### 反模式六：历史问题只写在缺陷系统里

若未转成运行时检测、人工门禁和回归测试，测试经验不会进入 Agent 交付质量控制。

### 反模式七：语义检查没有证据位置

只输出“这段可能有问题”会让用户重新通读全文，无法降低审查成本。

## 14. 落地验收清单

### 业务与结果

- [ ] 业务目标、输入、输出和支持范围明确；
- [ ] 交付物已拆成可定位的结果单元；
- [ ] 每个结果单元已标记关键性和验收标准；
- [ ] 关键依赖关系已记录。

### 历史问题

- [ ] 开发测试和用户自测问题已汇总；
- [ ] 每个重要问题已定义触发条件和影响结果；
- [ ] 每个问题有检测器或人工门禁；
- [ ] 每个问题有回归测试。

### 验证器

- [ ] 关键结果至少有一个确定性必要验证器；
- [ ] 验证器能够返回 PASS/FAIL/NOT_RUN/UNSUPPORTED/ERROR；
- [ ] 验证器失败能定位到具体对象；
- [ ] 验证器自身已测试；
- [ ] 语义验证器不是高可信的唯一依据。

### 等级规则

- [ ] 高可信要求所有必要验证通过；
- [ ] 未验证时不能判高；
- [ ] 关键失败能够一票降级；
- [ ] 不使用简单平均聚合；
- [ ] 中、低等级对应明确人工动作；
- [ ] 输入超出支持范围有明确处理。

### 可解释性与治理

- [ ] 等级能追溯到检查和证据；
- [ ] 输出包含风险位置和建议核查动作；
- [ ] 契约、风险、验证器和 Agent/Skill 版本关联；
- [ ] 变更时执行可信设计影响分析。
