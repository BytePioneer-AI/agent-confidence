# Agent 可信等级设计最佳实践

## 1. 适用范围

本实践用于指导开发人员针对具体业务 Agent + Skill 设计高 / 中 / 低可信等级。目标是把“全文逐项检查”转成“按风险位置定向检查”，同时避免把模型表达流畅或文本相似误认为业务正确。

## 2. 核心边界

本实践不规定统一运行时实现，不要求独立校验脚本、统一验证器接口、统一状态、统一聚合器或独立服务。可信判断逻辑可以直接融入现有 SQL、代码、API、规则引擎或工作流。

## 3. 核心定义

### 3.1 结果单元

可独立验收、定位和采取审查动作的最小业务结果，例如字段、记录、问卷题目与答案、测试用例、结果集合、段落、文件或业务决策。

### 3.2 可信含义

每个结果必须说明：

- 可信度评价什么；
- 高可信具体证明什么；
- 高可信不证明什么。

例如，SO 合同字段“高可信”可以表示字段来源明确、映射符合当前规则，但不表示源 SO 的业务录入绝对正确。

### 3.3 结果形态

常见形态：

- `scalar`：单值；
- `field`：字段或单元格；
- `record`：一条业务记录；
- `text`：文本答案或结论；
- `collection`：需要满足数量与覆盖维度的一组结果；
- `document`：完整文件；
- `decision`：路由或处理决策。

集合型结果不能只检查数量，还要检查必要维度、分支、边界、重复和缺失。

### 3.4 业务影响

可信等级和业务影响是两个维度。高可信但高影响的监管、法律、处罚、诉讼、财务金额等结果，仍可要求责任部门确认。

## 4. 设计流程

### 第一步：建立业务上下文

记录用户、输入输出、处理步骤、下游动作、错误影响、当前支持范围和权威数据源。

### 第二步：拆分结果单元

不要只给整个文件一个等级。拆到能定位具体字段、问题、测试用例、段落或步骤的粒度。

### 第三步：明确业务阶段

同一业务可能包含多个阶段，例如 ECS UAT 的规则理解、单条数据生成、测试集覆盖、系统导入和实际执行。不同阶段应分别设计可信条件。

### 第四步：整理历史问题

收集开发测试、用户自测、人工修正版和部门终版差异。对问题进行根因分类：

- 脚本实现错误；
- 规则文档缺失或不清晰；
- 新业务规则候选；
- 个案人工信息；
- 外部系统字段；
- 待业务确认。

### 第五步：判断是否可泛化

单个样例或人工修正版不天然等于通用规则。只有能够从业务输入稳定识别、适用于多个案例且完成业务确认的问题，才适合固化到可信条件或实现逻辑。

### 第六步：定义判断维度

根据业务选择维度，例如：

- 规则匹配；
- 主体一致性；
- 来源权威性；
- 时效性；
- 多来源一致性；
- 答案完整性；
- 单条结果正确性；
- 集合数量和覆盖度；
- 来源追溯；
- 下游系统验证。

### 第七步：定义硬性规则

硬性规则优先于加权分，例如：

- 问卷法人主体不一致时不得为高；
- 高影响合规结论无当前责任部门证据时不得自动填写；
- ECS 未完整匹配支持规则且关键条件无法解释时不得为高；
- 测试集合缺少 clean 或 dirty 核心场景时为低；
- SO 字段疑似使用错误来源或错误回退路径时为低。

### 第八步：定义高 / 中 / 低

#### 高

业务认为当前依据足以按约定动作使用。高的充分条件由业务决定，并应明确适用范围和不代表的内容。

#### 中

有可参考依据，但存在主体、时间、规则边界、来源、覆盖或客户口径问题，需要定向复核。

#### 低

缺少直接依据、关键条件冲突、候选来源不适用、结果集合严重缺失，或依赖模型推断高风险事实。通常应留空、批注、阻断或交责任部门。

### 第九步：设计证据链

每项结果尽量记录：

- 来源文件、表、行、字段或历史问题；
- 原始值和采用值；
- 适用主体、时间和客户；
- 使用的业务规则；
- 改写或转换说明；
- 需要检查的具体位置。

### 第十步：指定实现位置

建议将逻辑放到现有业务最自然的位置，例如转换代码、SQL、规则解析器、检索流程、写回流程、API 或人工审批节点。

## 5. 三种典型模式

### 5.1 映射与转换型

以 PBA SO 合同转换为代表。关注字段来源、直接映射、回退推导、规则命中、行级风险和 trace。

### 5.2 规则与结果集合型

以 ECS UAT 为代表。关注规则匹配、条件解析、单条 clean / dirty 正确性、数量、边界和分支覆盖。

### 5.3 历史知识复用型

以供应商问卷为代表。关注语义匹配、法人主体、当前时间、客户适用性、权威来源、部门职责和答案完整性。

## 6. 加权评分与概率

初期可以使用规则分帮助映射高 / 中 / 低，但必须：

- 先应用硬性规则；
- 不把内部规则分直接称为概率；
- 不用平均掩盖主体不一致或高风险问题；
- 后续可用人工终版是否实质修改进行离线校准。

## 7. 输出要求

每个结果至少应给出：

- 结果位置；
- 高 / 中 / 低；
- 判断依据；
- 来源路径；
- 业务影响；
- 建议审查动作；
- 适用范围或限制。

## 8. 反模式

- 让执行 Agent 仅凭主观自信打分；
- 用文本相似度直接代替答案可复用性；
- 只看生成数量，不看集合覆盖；
- 把同类型信息误认为当前主体的正确信息；
- 把单个用户修正直接写成通用规则；
- 用一个综合分掩盖高影响硬性问题；
- 只输出“请人工检查”，却不说明位置、原因和责任部门；
- 为了置信度额外建设与现有系统割裂的技术框架。

## 9. 落地检查清单

- [ ] 业务目标、范围和下游动作明确；
- [ ] 已拆分结果单元与业务阶段；
- [ ] 每项结果定义了可信含义和限制；
- [ ] 已识别结果形态和集合覆盖要求；
- [ ] 历史问题已归因并判断可泛化性；
- [ ] 高、中、低条件由业务确认；
- [ ] 关键硬性规则未被加权分掩盖；
- [ ] 来源、主体、时间和适用性可追溯；
- [ ] 业务影响和可信等级分开；
- [ ] 已明确逻辑在现有系统中的实现位置；
- [ ] 人工审查动作能够定位具体对象。
