# Agent 业务结果置信度设计实践

## 1. 目标

针对 Agent + skill 生成的业务结果，建立结果级的高、中、低可信等级，使用户能够：

- 对高可信结果直接使用，或根据业务影响和既有业务要求进行抽查或复核；
- 对中可信结果只检查已经定位的结果及其来源；
- 对低可信结果进行人工确认、修正或阻断；
- 通过结果溯源链快速验证结果，而不必审查全部输入和处理过程。

置信度设计的目标是减少用户对业务结果的审查工作量，并让仍需进行的审查更集中、更容易。

## 2. 适用范围与边界

本方法适用于：

- 信息提取和字段填写；
- 数据清洗、转换、计算和汇总；
- Excel、Word 等文档内容生成；
- 问卷回答、摘要和报告生成；
- 分类、判断、结论和建议；
- 同时包含结构化结果和自然语言结果的业务任务。

本方法关注：

- 业务真正要求输出什么；
- 输出应拆成哪些结果单元；
- 每个结果单元如何追溯到实际来源；
- 每个结果单元在什么业务条件下属于高、中、低；
- 中、低可信结果需要用户检查什么。

本方法不处理：

- 整个文件或整个任务的总体置信度；
- 把多个结果单元平均或合成一个总体等级；
- 文件能否打开、模板结构是否完整等交付校验；
- 应输出内容是否全部生成等任务完成度检查；
- 输入、来源、中间数据或处理步骤自身的置信度；
- 没有业务校准依据的百分比分数；
- 模型对自身结果的主观自信。

## 3. 核心模型

业务结果置信度围绕结果单元建立：

```text
业务需求
→ 明确业务输出
→ 先列出可能的结果单元
→ （已有业务实现时）分析当前实际结果溯源链：结果是什么、来源在哪里、来源怎样形成结果
→ 确认最终的结果单元（已有业务实现时依据当前实际链校验）
→ 明确每个结果需要保留哪些溯源信息
→ 根据业务验收标准和具体溯源信息定义高、中、低，并按需记录业务影响
→ 说明中、低结果具体怎么检查
```

如果业务 Skill、脚本或工作流已经实现，应先逐类查清当前形成了什么结果、实际使用了哪些来源，以及来源怎样形成结果。这些信息用于校验结果单元并设计置信度，不需要复述整个技术流程。

如果尚无已实现的业务流程，则跳过现状分析，不猜测结果当前如何形成，也不代替业务 Skill 设计处理流程。

完整的 Excel、Word、报告或数据包只是用来汇集和呈现多个结果，不是另一个需要评定总体置信度的结果单元。

每个结果单元至少包含以下设计信息：

```text
业务含义
+ 结果位置
+ 业务验收标准
+ 业务影响（可选）
+ 必须保留的结果溯源信息
+ 高、中、低条件
+ 中、低结果的具体检查方式
```

### 3.1 结果单元

结果单元是：

> 业务要求输出中，用户能够独立接受、退回或修正的最小业务对象。

结果单元没有固定的数据形态。它可以是：

- 问卷中的一道问题答案；
- Word 模板中的一个字段或内容块；
- Excel 中的一个业务指标或一条记录；
- 一个分类结果；
- 一个业务判断、事实、结论或建议；
- 其他需要独立验收的业务输出。

结果单元必须属于业务要求的输出。输入、来源、处理记录和中间产物不能仅因为“可以检查”就成为结果单元。

### 3.2 结果单元应该拆分到什么程度

判断是否需要拆成多个结果，主要看用户是否需要分别接受、退回或修改它们：

- 用户可以分别接受、退回或修正的内容，应分别定义结果单元；
- 用户只能整体确认或修改的内容，不必继续拆成更细的事实；
- 来源更多，不代表结果单元更多；
- 处理步骤更多，不代表结果单元更多；
- 不追求字段、句子或事实级的无限细分。

例如，一道问卷答案引用三个历史来源，但用户仍以整道答案为单位确认，那么它仍是一个结果单元。只有答案中的不同部分需要分别验收或修正时，才继续拆分。

### 3.3 哪些中间内容不算结果

处理过程中产生的临时文件、清洗数据、候选列表或处理记录，默认不属于业务结果。它们通常只是结果的来源或实现过程中的辅助信息。

只有同时满足以下条件时，中间产物才转为结果单元：

- 它属于业务明确要求的输出；
- 用户需要独立使用或验收它；
- 它可以被单独接受、退回或修正。

例如，生成月度销售额之前形成的标准化明细，如果只是内部计算数据，应进入月度销售额的结果溯源链；如果标准化明细本身也是需要交付和验收的业务输出，才将其定义为结果单元。

### 3.4 结果溯源链

每个结果单元都必须具有可追溯的结果溯源链。

结果溯源链记录结果实际来自哪里，以及来源经过怎样的实际处理形成了结果。它的最小内容是：

```text
一个结果位置
+ 至少一个来源位置
+ 明确的来源—结果形成关系
```

在已有业务实现时，需要区分：

- **当前实际结果溯源链**：说明当前实现实际怎样形成结果；
- **每个结果需要保留的溯源信息**：说明为了让用户核查结果，需要记录和展示哪些来源、处理方式和结果位置。

前者只记录能从现有实现和运行记录中确认的事实，后者是以后每次运行都要满足的长期要求。不能把长期要求写成当前已经具备的事实。具体的现状分析方法见第 5 章。

来源位置可以定位到：

- 文件、工作表、行、列或单元格；
- Word 的章节、段落、表格或字段；
- 数据库表、记录和字段；
- API 返回记录；
- 已确认的业务规则、计算口径或用户要求。

来源—结果形成关系可以是：

- 原样提取；
- 格式或单位转换；
- 字段映射；
- 公式计算；
- 规则判断；
- 相似问题匹配；
- 多来源汇总；
- 内容合并。

结果溯源链不要求固定的线性结构。一个结果可以由一个来源直接形成，也可以由多个来源共同形成。只有实际发生处理时，才记录对应的转换、计算或规则。

现有业务流程可以帮助查明当前实际结果溯源链，但完整业务流程不等于某个结果的溯源链。结果溯源链只记录与该结果有关的来源、实际处理和结果位置；共用的文件扫描、候选缓存或临时处理只有在实际参与形成该结果时才需要记录。

结果溯源链是事实记录，不定义高、中、低，也不用于汇总出其他等级。来源是否过期、是否适用于当前业务或是否彼此冲突，应直接用于判断结果的置信度，不要另设“来源置信度”。

结果溯源链不记录模型内部思维过程，也不能把模型自述的解释当作证据。

### 3.5 业务影响

业务影响是可选维度。如果不同结果出错后造成的影响不一样，或者业务需要区分哪些结果应优先、重点审查，就可以使用这个维度；否则可以不定义。

业务影响可以说明结果错误可能造成的具体后果，例如：

- 财务损失；
- 法律或合规风险；
- 错误业务决策；
- 对外披露错误；
- 返工或人工核查成本；
- 低影响的表达或体验问题。

业务影响与置信度是两个不同维度：

- **置信度回答“用户是否需要审查这个结果”；**
- **业务影响回答“如果结果错误，会造成什么后果”。**

本方法不统一规定业务影响等级。业务影响应优先记录具体错误后果，避免再建立一套与高、中、低置信度容易混淆的通用分级。

如果具体业务已经存在正式的风险、严重度或影响分类，应直接复用；没有现成分类时，可以不定义业务影响等级，也可以根据实际需要自行定义。

业务影响不改变结果的高、中、低置信度，但可以独立影响审查优先级和审查强度。例如：

| 结果单元 | 置信度 | 业务影响 | 审查方式 |
|---|---|---|---|
| 合同付款金额 | 高 | 如果仍然出错，可能造成重大财务损失 | 虽然结果依据充分，仍按财务控制要求仔细复核 |
| 一般说明性字段 | 中 | 错误只造成局部文字修改，不影响主要业务判断 | 只需快速核对标记内容及其来源 |

需要分级的业务可以参考下面的示例，但不要求采用这些名称、等级数量或判断条件：

| 示例等级 | 示例定义 | 示例处理方式 |
|---|---|---|
| 关键 | 错误可能造成重大财务、法律、合规或关键决策后果 | 即使置信度为高，也可以要求强制复核 |
| 重要 | 错误会影响主要业务内容或关键下游动作，但通常可以修正 | 优先处理需要审查的结果 |
| 一般 | 错误影响局部内容，不改变主要业务判断 | 根据置信度进行常规或快速审查 |

该分级只是业务影响维度的示例，不是通用要求。如果置信度和现有检查方式已经足以满足业务需要，可以完全不使用业务影响维度。

### 3.6 高、中、低分别表示什么

| 等级 | 通用语义 | 默认用户动作 |
|---|---|---|
| 高 | 满足该业务定义的直接使用条件 | 无需因为置信度专门检查，或按业务策略少量抽查 |
| 中 | 未确认存在错误，但有明确的不确定点或边界条件 | 只检查该结果及已定位的溯源信息 |
| 低 | 已发现错误，或命中不可接受的重大风险 | 人工确认、修正或阻断后续使用 |

高、中、低只属于业务结果单元。具体条件必须根据业务验收标准定义，不能仅根据内容是否由模型生成来判断。

表中的用户动作只表示由置信度触发的默认审查要求。业务仍可根据可选的业务影响维度或既有控制制度，对高可信的关键结果要求额外复核。

“模型生成”是结果形成方式，不是置信等级。模型生成的结果仍可能根据业务条件分别属于高、中或低。

## 4. 核心原则

### 4.1 从业务输出开始

先明确业务要求的输出以及用户如何使用和审查这些输出，再初步识别候选结果单元。不要从技术流程、Agent 步骤、中间文件或日志反推结果单元。

已有业务实现时，再用当前实际生成的结果校验候选结果单元。结果溯源链中的来源和处理步骤不能因此变成新的结果单元。

### 4.2 每个结果都必须可追溯

每个结果单元必须能够定位到实际来源及其形成关系。无法建立结果溯源链的结果不能定义为高可信。

是否将其定义为中或低，应根据该业务对可追溯性和结果风险的要求决定。

### 4.3 只评定结果，不评定结果溯源链

输入、来源、处理记录和其他辅助信息用于判断和验证结果，但不建立独立置信等级。

用户审查的对象始终是业务结果。结果溯源链帮助用户直接找到核对该结果所需的来源和处理记录。

### 4.4 可信条件必须针对业务

通用方法只统一高、中、低的语义，不替业务负责人定义正确性。

例如：

- 财务金额可能要求来源记录、币种、期间和计算口径严格一致；
- 合同日期可能要求与签署文件的明确条款一致；
- 问卷答案可能要求历史来源的主体、年份和适用范围一致；
- 内部说明文案可能允许不影响事实的措辞差异。

### 4.5 不定义整体置信度

不要给完整文件或整个任务计算一个总体的高、中、低，也不要机械平均多个结果单元。

用户应直接看到哪些结果可以使用、哪些结果只需检查指定内容、哪些结果必须修正。

### 4.6 中、低结果必须说明具体怎么检查

具体检查方式必须说明：

- 检查哪个结果；
- 查看哪些来源位置；
- 核对什么业务条件；
- 检查后如何处理。

避免只写“请人工检查”。

### 4.7 由最容易取得准确数据的环节判断置信度

每条置信度规则需要哪些数据，就尽量由最容易取得这些准确数据的现有环节来判断。例如，汇总公式可以由生成脚本复算，业务状态可以由权威 API 判断。不要把所有判断都交给大模型，也不要为了形式统一再建设一套重复系统。

大模型可以帮助判断两段文字含义是否一致、发现内容冲突，或检查现有来源是否足以支持结果，但不能通过评价自己的回答来认证结果正确。

## 5. 推荐设计流程

### 第一步：建立业务背景

记录：

- 业务目标；
- 使用者和下游动作；
- 明确输入；
- 明确输出；
- 业务验收标准；
- 支持范围和不支持范围；
- 结果错误可能造成的影响（如业务需要单独管理）。

### 第二步：先列出可能的结果单元

从业务输出中先列出可能的结果单元类型，并记录：

| 内容 | 说明 |
|---|---|
| 结果单元类型 | 例如单个问卷答案、合同日期、月度销售额 |
| 业务含义 | 该结果解决什么业务问题 |
| 识别和定位方式 | 如何在输出中找到该结果 |
| 用户确认和修改的单位 | 用户是逐字段、逐行还是按整段内容接受、退回或修改 |
| 业务验收标准 | 什么叫正确、完整和可用 |
| 业务影响（可选） | 结果错误会造成什么后果；仅在业务需要单独管理时记录 |

如果一个候选对象只是输入、来源、缓存、候选集、转换数据、处理日志或执行步骤，应把它记录为来源或实现细节，不要放进结果单元清单。

### 第三步：已有实现时，了解结果实际如何形成

这一步仅适用于业务 Skill、脚本或工作流已经实现的场景。应查看现有实现、代表性输入输出，以及必要的运行记录。

针对每类候选结果，单独记录：

- 当前实现是否实际形成该结果，以及结果写在哪里；
- 实际使用了哪些来源，分别位于哪里；
- 来源与结果之间实际发生了什么关系；
- 当前已经记录了什么、用户能查看什么，以及哪些内容无法确认。

现状分析可以按结果类型和不同形成方式总结，不需要在设计文档中穷举每次运行的所有结果。但要确认当前实现能否为每个具体结果保留实际来源、形成关系和结果位置。可能使用但实际未采用的候选来源，不能写成实际来源。

查不到的内容标记为“未知”，不要按理想流程补齐。尚无业务实现时跳过这一步，不猜测或设计业务处理流程。

### 第四步：确认最终的结果单元

已有业务实现时，根据第三步的分析校验候选结果单元：

- 当前实现是否真正产生了可定位的业务结果；
- 用户是否能按当前划分方式独立接受、退回或修正该结果；
- 候选对象是否只是来源、处理节点、临时数据或人工待办；
- 是否把应该一起确认的内容错误拆开，或把可以分别确认的内容错误合在一起。

当前尚未生成、仅留空或待人工完成的内容，不能写成当前实现已经生成的结果。如果它未来仍是必须交付的业务输出，可以继续为它设计置信度，但要明确标注当前尚未实现。没有业务实现时，只根据已明确的业务输出和验收方式确认结果单元。

### 第五步：明确每个结果必须保留哪些溯源信息，并检查当前实现还缺什么

针对每类结果单元，定义：

- 必须记录哪些来源，以及来源需要精确到什么位置；
- 如何记录来源形成结果的关系，包括多来源结果如何关联全部实际来源；
- 用户从结果位置如何查看相应溯源信息。

已有业务实现时，再检查当前是否已经记录和展示这些信息，列出还缺少的内容。当前实际情况、以后每次运行必须保留的信息和当前缺项不能混写。

业务置信度定义只保存已确认的长期要求。某次任务实际使用的来源和结果位置，属于该次任务的运行记录。

### 第六步：说明什么情况是高、中、低

为每类结果直接写业务条件，不使用抽象分数。

高、中、低的判断条件只能使用用户或程序能够检查的具体事实，例如实际来源、适用主体、数据年份和计算规则。已有业务实现尚未记录某项必要事实时，应明确写出缺少什么，不能假定当前实现已经能够使用这项事实判断高可信。

至少回答：

- 什么条件满足时可以直接使用；
- 什么不确定点需要用户检查指定的结果和来源；
- 什么明确错误或重大风险必须修正或阻断；
- 结果溯源中的哪些事实会影响结果等级。

例如，单个问卷答案可以定义为：

| 等级 | 示例条件 |
|---|---|
| 高 | 答案有明确历史来源；来源问题与当前问题含义一致；主体、年份和适用范围满足当前要求 |
| 中 | 有可参考来源且未发现明确冲突，但来源年份、主体或口径需要确认 |
| 低 | 答案与来源内容冲突；引用错误来源；将不适用的历史事实作为当前确定事实 |

### 第七步：说明中、低结果具体怎么检查

分别说明中、低结果要检查什么。检查说明应直接指出结果位置和对应的结果溯源链。

例如：

- 检查当前问卷第 12 题答案，并对照历史问卷 `Sheet2!A37:B37` 的主体和年份；
- 核对合同结束日期结果，并查看源合同第 4 页第 8 条；
- 检查净销售额，并依据已列出的订单、退款记录和计算口径复算。

### 第八步：确定由哪个环节判断置信度

针对每条高、中、低条件，明确由哪个现有环节取得所需数据并完成判断：

- SQL 或数据任务；
- 文件解析和生成脚本；
- 现有业务代码；
- 权威业务 API；
- 规则引擎；
- 工作流节点；
- 人工审核节点。

同时说明判断需要哪些数据、结果溯源信息由哪里记录，以及中、低结果通过什么方式展示给用户。

设计与代码修改是两个不同动作。是否修改实际脚本、规则或业务代码，应根据项目安排和用户授权决定。

## 6. 典型场景

### 6.1 供应商调查问卷

业务输出是一份填写完成的问卷，但结果单元通常是每道需要独立确认的答案。

ps：如果审核人员或审核部门分配也需要被独立验收，可以将其定义为另一类结果单元。

```text
历史问卷中的问题和答案
──[相似问题匹配、必要的主体或年份调整]──>
当前问卷中的单题答案
```

以下内容默认不是结果单元：

- 知识库记录；
- 检索候选列表；
- 匹配日志；
- 临时 JSON；
- 答案写回步骤。

它们可以成为单题答案的来源或实现记录，但不拥有独立置信等级。

### 6.2 从多个文件填写 Word 模板

完整 Word 只是用来汇集和呈现结果，每个需要独立确认的填写字段或内容块才是结果单元。

```text
源文件中的合同日期
──[日期提取、格式标准化]──>
模板中的“合同结束日期”字段
```

如果一个内容块由多个文件共同形成，结果溯源链应关联全部实际来源。

### 6.3 数据计算和汇总

如果月度销售额是业务需要使用的指标，它是结果单元：

```text
订单明细 + 退款记录 + 已确认计算口径
──[汇总计算]──>
月度净销售额
```

计算过程中形成的标准化数据，如果不是业务要求独立交付和验收的输出，应作为结果溯源或实现数据。

## 7. 测试问题和用户反馈管理

开发测试、用户自测和验收反馈可以帮助完善置信度定义。这些问题和反馈必须来自真实记录，不能凭空假设，也不能在用户确认前直接改变当前规则。

推荐流程：

```text
用户提供真实问题或反馈
→ 保留原始描述并归档
→ 分析可能影响的结果单元和置信度条件
→ 形成修改建议
→ 用户确认
→ 更新当前业务置信度定义
```

已经确认的问题应转化为对应结果单元的高、中、低条件或具体检查方式；原始问题及处理过程保留在反馈档案中。

建议在业务项目根目录维护：

```text
.agent-confidence/
├── definition.md
└── feedback.md
```

- `definition.md`：只保存用户已经认可的当前业务置信度定义和结果溯源要求；
- `feedback.md`：追加保存真实测试问题、用户反馈、分析建议、用户决定和最终处理结果。

未确认的分析建议必须与 `definition.md` 分开。当前实现的分析结果、实现缺项和某次任务的具体结果溯源链，也不写入这两个业务级文件。

## 8. 开发阶段与运行阶段

业务置信度定义属于开发和维护阶段的规范。运行阶段的置信度通常由脚本、SQL、业务代码、API、规则或人工审核节点根据已确认条件判断。

业务置信度定义不要求业务运行流程动态读取，也不要求使用统一的运行时协议、固定状态枚举、通用验证器接口或独立置信度服务。

运行时向用户呈现的信息应至少能够回答：

- 哪个结果被评定；
- 结果是高、中还是低；
- 中、低结果为什么需要审查；
- 用户应检查什么；
- 如何查看对应的结果溯源链。

具体展示在 Excel 批注、辅助工作表、Word 批注、页面、JSON、日志或其他位置，应由业务需要决定。

## 9. 设计审查清单

上线或变更前检查：

- 结果单元是否都来自业务明确要求的输出；
- 是否把输入、中间数据或关键处理步骤误当成结果单元；
- 结果单元的划分是否符合用户实际接受、退回或修正内容的方式；
- 每个结果单元是否有明确位置和业务验收标准；
- 如采用业务影响维度，是否记录了具体错误后果，并避免与高、中、低置信度混淆；
- 已有业务实现时，是否查清了结果位置、实际来源和形成关系，并把查不到的内容标记为“未知”；
- 是否根据当前实际生成的结果校验了候选结果单元，没有把来源、中间内容或尚未生成的内容当作当前结果；
- 是否明确了每个结果必须保留的溯源信息；已有实现时，是否另外列出当前还缺少什么；
- 是否错误地为来源或处理节点定义了置信度；
- 是否为每类结果定义了具体的高、中、低条件；
- 中、低结果是否明确说明了要检查的位置、内容和检查后的处理方式；
- 是否错误地定义了完整文件或整个任务的总体置信度；
- 是否使用模型主观自信或“是否由模型生成”替代业务条件；
- 是否只在收到真实测试问题或用户反馈后才建立问题记录；
- 未确认建议是否与当前生效定义保持分离；
- 每条置信度规则是否由最容易取得所需准确数据的现有环节判断。

## 10. 不推荐做法

- 把输入、来源、候选记录、中间数据或处理步骤作为结果单元；
- 因为一个对象可以检查，就为它定义独立置信度；
- 把未核实或尚未生成的内容写成当前事实；
- 为结果溯源链中的每个节点定义高、中、低；
- 要求用户逐个检查结果溯源链中的所有节点；
- 把模型内部思维过程作为结果证据；
- 让 Agent 根据自己的主观把握直接给结果评级；
- 把“模型生成”作为与高、中、低并列的置信等级；
- 用一个百分比分数覆盖不同业务结果；
- 给整个文件或任务计算总体置信度；
- 机械平均多个结果单元；
- 只写“需要人工检查”，却不说明结果位置、来源位置和核查动作；
- 猜测不存在的测试问题或用户反馈；
- 未经用户确认就修改当前业务置信度定义；
- 为了形式统一，强制所有业务采用同一组运行状态、接口或计算框架。
