# Agent 可信等级设计最佳实践

## 1. 目标

针对 Agent + Skill 的业务产出，建立高、中、低可信等级，使用户能够：

- 对高可信结果直接使用或少量抽查；
- 对中可信结果只检查指定区域；
- 对低可信结果进行人工确认或阻断；
- 了解风险来自哪个结果、哪类问题以及应采取什么核查动作。

本实践关注“如何分析和定义可信等级”，不规定统一运行时技术框架。

## 2. 适用范围

适用于：

- 数据处理、清洗、汇总；
- Excel 生成、公式和图表；
- Word 报告、摘要和分析；
- 信息提取和字段映射；
- 业务规则执行；
- 多步骤 Agent 工作流；
- 同时包含结构化结果和自然语言结果的任务。

## 3. 核心原则

### 3.1 先定义结果，再定义可信等级

不要直接给整个文件或整个任务一个模糊等级。先拆出可以独立验收和独立审查的结果单元，例如：

- 数据记录或字段；
- 指标或汇总值；
- Excel 公式、图表或工作表；
- Word 中的数字、事实、结论或建议；
- 一个业务决策；
- 一个关键处理步骤。

拆分粒度以“能够产生具体人工核查动作”为准。

### 3.2 可信等级必须针对业务定义

不同业务对高、中、低的条件不同。通用方法只能统一设计过程，不能替业务负责人定义正确性。

例如：

- 财务金额可能要求明细与总额严格勾稽；
- 内部文案可能允许轻微措辞差异；
- 合同日期、付款金额和合规结论通常属于关键结果；
- 排版颜色和非关键标题通常属于低影响结果。

### 3.3 测试问题是可信设计的重要输入

开发测试和用户自测中确认的问题，应经过整理后进入可信设计。

每个问题至少说明：

- 问题是什么；
- 在什么条件下发生；
- 影响哪些结果；
- 严重程度；
- 能否在现有系统中识别；
- 命中后应落到哪个可信等级；
- 用户应检查什么。

不要把零散主观反馈直接当成等级条件。先归纳成可描述的业务风险。

### 3.4 实现形式不做统一限制

可信判断逻辑可以放在：

- 现有 SQL；
- ETL 或数据任务；
- Python、Java、JavaScript 等处理代码；
- 规则引擎；
- 业务 API；
- Agent 工具调用或工作流节点；
- 文档生成后的检查步骤；
- 人工审批节点。

通用 Skill 只需要说明建议实现位置和实现方式，不要求创建独立“置信度脚本”。

### 3.5 大模型适合分析和补充语义风险，不适合自我认证

大模型可以帮助：

- 拆分结果；
- 归类历史问题；
- 发现语义矛盾；
- 判断结论是否缺少数据支持；
- 生成落地建议和审查清单。

但不应仅因为 Agent 自称“有把握”，就将结果定义为高可信。

### 3.6 整体等级不能掩盖关键结果

当结果包含多个子结果时，应优先考虑关键性和依赖关系。

例如：

- 付款金额为低，即使其他 99 个非关键字段为高，整体也不能被平均为高；
- 上游清洗数据存在高风险时，依赖它生成的指标和报告结论也应受到影响；
- 非关键排版问题通常不应把整个业务结果直接降为低。

## 4. 推荐设计流程

### 第一步：建立业务背景

记录：

- 业务目标；
- 使用者；
- 输入；
- 处理过程；
- 输出；
- 下游动作；
- 错误可能造成的影响；
- 支持范围与不支持范围。

### 第二步：拆分结果单元

为每个结果记录：

- 名称和示例；
- 所在位置；
- 关键性；
- 上游依赖；
- 业务验收标准；
- 错误后果。

推荐关键性分为：

- `critical`：错误可能造成重大业务、财务、法律或合规后果；
- `high`：影响主要业务结论或关键下游动作；
- `medium`：需要修正，但一般不会直接造成严重后果；
- `low`：主要影响格式、表达或体验。

### 第三步：整理已知问题

建议使用下表：

| 字段 | 说明 |
|---|---|
| 问题编号 | 便于追踪 |
| 来源 | 开发测试、用户自测、验收测试等 |
| 问题描述 | 实际发生了什么 |
| 触发条件 | 哪些输入或场景下发生 |
| 影响结果 | 哪些结果可能错误 |
| 严重度 | critical / high / medium / low |
| 可识别方式 | SQL、代码、API、规则、人工等 |
| 等级影响 | 命中后对应高、中、低中的哪一级 |
| 核查动作 | 用户具体检查什么 |

### 第四步：定义每个结果的高、中、低

建议为每个结果直接写条件，而不是写抽象分数。

示例：月度销售额

| 等级 | 业务条件 |
|---|---|
| 高 | 输入文件范围明确；日期和金额处理符合业务口径；月度金额可由明细独立复算；报告引用值与汇总值一致 |
| 中 | 核心金额未发现明确冲突，但输入属于边界格式、来源映射不完整或部分月份需要定向核对 |
| 低 | 明细与汇总不一致；关键文件遗漏；金额口径错误；命中已知严重缺陷 |

示例：经营原因分析

| 等级 | 业务条件 |
|---|---|
| 高 | 结论有直接业务数据或明确来源支持，且没有与事实指标矛盾 |
| 中 | 有间接迹象，但因果依据不足，需要人工判断 |
| 低 | 与数据矛盾、引用错误事实或将无依据推测写成确定结论 |

### 第五步：定义整体组合原则

组合原则应由业务关键性驱动，常见规则包括：

- 任一 `critical` 结果为低，整体为低；
- 任一 `critical` 结果为中，整体最高为中；
- 高风险上游结果会限制下游结果的最高等级；
- 多个中风险非关键结果可以使整体降为中；
- 低影响格式问题通常不单独决定整体等级。

不要机械平均多个子结果。

### 第六步：设计实现位置

针对每条可信条件，回答：

- 现有系统中哪里最容易获得判断所需信息；
- 是否可以复用已有 SQL、测试、日志或 API；
- 是否需要增加中间数据、来源映射或处理记录；
- 逻辑适合在生成前、生成中还是生成后执行；
- 命中风险后如何影响当前流程；
- 需要向用户展示什么位置和原因。

示例：

| 判断条件 | 推荐实现位置 | 推荐方式 |
|---|---|---|
| 明细与汇总一致 | 原有数据处理任务 | SQL 或代码独立复算 |
| Word 数字与 Excel 一致 | 报告生成后 | 文档解析 + 数据比对 |
| 输入文件是否齐全 | 任务入口 | 文件清单或任务元数据比对 |
| 因果结论是否有依据 | 分析生成阶段或审查阶段 | 数据来源约束 + 语义辅助检查 |

### 第七步：定义人工审查策略

中、低等级必须转换成具体动作，例如：

- 核对“销售汇总”工作表 D42 的公式范围；
- 检查经营报告第 3 段中的增长率是否使用同比口径；
- 确认合同结束日期是否为 2027-03-31；
- 核查原因分析是否有客户订单或流失数据支持。

避免只写“请人工检查”。

## 5. 高、中、低的通用解释

### 高

该业务为此结果定义的高可信条件均满足，可以直接使用或按业务策略少量抽查。

### 中

结果存在边界场景、依据不足、历史风险或需要业务判断的部分，应只检查系统标记的位置。

### 低

命中明确错误条件、严重风险场景或关键业务要求不满足，应人工确认或阻断后续流程。

这三种解释只提供统一语义，具体条件由每个业务定义。

## 6. Skill 输出建议

一份业务可信设计至少应包含：

1. 业务背景和范围；
2. 结果单元清单；
3. 已知问题和风险映射；
4. 各结果的高、中、低条件；
5. 整体等级组合原则；
6. 判断逻辑的建议实现位置；
7. 定向人工审查动作；
8. 未确认的业务问题；
9. 变更后需要重新评估的内容。

## 7. 设计审查清单

上线前检查：

- 是否只给整个文件定义了一个粗粒度等级；
- 是否遗漏关键结果；
- 是否说明每个结果的高、中、低条件；
- 是否覆盖开发测试和用户自测中的重要问题；
- 是否把关键错误与低影响格式问题区分开；
- 是否说明判断逻辑融入现有系统的位置；
- 是否能定位人工核查区域；
- 是否存在用大量非关键结果掩盖关键失败的情况；
- 是否明确哪些业务问题仍需负责人确认；
- Agent 或业务规则变化后，是否同步更新可信设计。

## 8. 不推荐做法

- 让 Agent 对自己的结果直接给出主观置信等级；
- 用一个百分比分数覆盖所有业务；
- 所有业务强制采用同一组运行状态和接口；
- 为了置信度另建一套与现有系统割裂的重复逻辑；
- 只统计整个 Agent 的平均正确率；
- 只写“需要人工检查”，却不说明检查位置和动作；
- 把测试问题停留在缺陷列表，而不转成等级条件和实现建议。
