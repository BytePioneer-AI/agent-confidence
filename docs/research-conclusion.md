# 调研结论：通用方法与业务实现边界

## 1. 最终结论

Agent 可信等级的通用成果应当是：

1. **最佳实践文档**；
2. **通用设计与审查 Skill**；
3. **轻量设计模板和静态检查工具**。

具体可信等级判断逻辑由每个业务根据实际情况实现，并可以直接融合到已有 SQL、代码、API、前端或 Agent 工作流中。

## 2. 不建设统一运行时框架

当前方案不要求：

- 所有业务编写独立校验脚本；
- 所有业务实现统一验证器接口；
- 所有业务采用统一运行状态；
- 所有业务调用统一聚合器；
- 所有业务建设置信度服务；
- 所有业务把判断逻辑从现有代码中拆出。

原因是不同业务的结果、数据来源、验收标准和实现方式差异很大。

## 3. 通用 Skill 的准确定位

`agent-confidence-designer` 用于帮助开发人员：

- 理解业务背景；
- 拆分结果单元；
- 定义每项结果的可信含义；
- 识别结果形态和业务阶段；
- 汇总开发测试和用户自测问题；
- 对问题进行归因、泛化和业务确认；
- 选择可信判断维度；
- 定义硬性条件和高、中、低；
- 定义完整性、数量和覆盖要求；
- 设计来源追溯；
- 分离可信等级与业务影响；
- 建议判断逻辑在现有系统中的实现位置；
- 生成定向人工审查动作。

## 4. 三个实际案例的验证

### SO 合同转换 Agent

说明可信度逻辑可以直接嵌入现有转换代码，并在字段、行、Excel 标色和 trace 中体现。

### ECS UAT 自动化 Agent

说明可信度不仅取决于规则是否匹配，还取决于单条测试数据正确性，以及测试结果集的数量、分支和边界覆盖。

### 供应商调查问卷整合

说明文本相似度不能等同于答案可信度，主体、时间、来源权威性、答案完整性和来源路径是关键条件。

## 5. 通用方法的核心

通用的不是计算公式，而是以下分析步骤：

```text
可信对象
→ 可信含义
→ 结果形态和业务阶段
→ 历史问题归因与泛化
→ 判断维度与硬性条件
→ 高 / 中 / 低
→ 来源路径和完整性
→ 业务影响与审查动作
→ 现有系统中的实现位置
```

## 6. 规则分和概率

业务可以使用内部规则分辅助划分高中低，但没有历史终版校准前，不应把规则分解释为概率。

后续可以利用：

- 初稿是否被原样采用；
- 是否仅格式调整；
- 是否发生实质修改；
- 是否由人工补充；
- 是否被删除；

来检验高中低规则的实际业务含义。

## 7. 最终架构表达

> 最佳实践负责统一思考方式，通用 Skill 负责把方法应用到具体业务，业务团队负责决定高中低条件并将判断逻辑融入现有系统。
