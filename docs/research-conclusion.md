# 调研结论：通用能力与业务定制边界

## 结论

面向异构业务 Agent 的可信等级体系，适合沉淀为两项通用成果：

1. **最佳实践文档**：统一术语、原则、流程、等级定义、职责边界和反模式；
2. **通用设计与审查 Skill**：辅助开发人员为具体 Agent + Skill 生成和审查业务专属的可信等级方案。

具体的运行时校验逻辑应由各业务根据数据结构、验收标准和已知缺陷，编写固定脚本实现。不存在一个无需业务上下文、能够可靠计算所有 Agent 结果可信等级的通用脚本。

## 为什么运行时校验必须业务定制

不同 Agent 的“正确”含义不同：

- Excel 汇总任务需要验证字段映射、行数、公式、明细与汇总勾稽；
- Word 报告任务需要验证数字来源、章节完整性、事实一致性和结论依据；
- 合同任务需要验证条款、金额、日期、主体和例外条件；
- 数据迁移任务需要验证记录守恒、主键、转换规则和落库结果。

这些检查依赖具体输入、输出、业务口径和系统接口，无法由通用层预先实现。

## 通用 Skill 的定位

通用 Skill 不直接判断某次运行结果是高、中还是低。它负责回答：

> 针对当前业务 Agent，应当拆分哪些结果、检查哪些风险、实现哪些验证器，以及如何依据验证结果确定高、中、低？

其主要输出包括：

- 可信等级契约；
- 已知风险模式库；
- 验证器规范；
- 业务校验脚本骨架；
- 回归测试要求；
- 设计覆盖度与缺口报告。

## 最终架构

> 通用文档定义原则，通用 Skill 生成并审查方案，业务固定脚本执行检查。

运行时可以复用统一的验证结果格式和等级聚合原则，但业务检查本身必须定制。
