# 调研结论：通用成果与业务落地边界

## 1. 最终结论

Agent 可信度体系适合沉淀为两项通用成果：

1. **最佳实践文档**：规定如何分析业务、拆分结果、沉淀测试问题、定义高/中/低及设计人工审查策略；
2. **通用设计 Skill**：在具体 Agent + Skill 场景中，按统一流程生成或审查业务专属可信等级设计。

具体可信判断逻辑必须结合业务实现，但不一定是独立脚本。它可以融合到已有 SQL、代码、API、工作流或人工审核节点中。

## 2. 为什么不做通用运行时框架

不同业务的差异不仅体现在数据结构，还体现在：

- 什么结果需要判断；
- 什么叫正确；
- 哪些错误影响最大；
- 可使用什么权威数据源；
- 判断逻辑运行在哪里；
- 结果如何被下游使用。

因此，当前通用方案不应强制：

- 固定的运行时状态枚举；
- 标准验证器接口；
- 独立置信度计算服务；
- 所有业务使用同一个聚合程序；
- 统一的百分比置信分。

## 3. 置信度设计通用skill 的定位

置信度设计通用skill 的职责是指导开发人员：

```text
理解业务
→ 拆分结果
→ 汇总测试问题
→ 提炼风险条件
→ 定义各结果的高/中/低
→ 设计整体组合原则
→ 建议融入现有系统的位置和方式
→ 形成最小人工审查清单
```

它不直接判断某次运行结果，也不替代业务代码。

## 4. 通用与定制的边界

| 内容 | 是否通用 |
|---|---|
| 业务分析方法 | 通用 |
| 结果拆分方法 | 通用 |
| 测试问题结构化方法 | 通用 |
| 高、中、低的通用语义 | 通用 |
| 设计审查清单 | 通用 |
| 某业务的具体等级条件 | 业务定制 |
| SQL、代码或 API 判断逻辑 | 业务定制 |
| 判断逻辑在现有系统中的位置 | 业务定制 |
| 人工审查阈值和阻断条件 | 业务定制 |

## 5. 推荐交付物

最终调研成果建议包括：

- 《Agent 可信等级设计最佳实践》；
- `agent-confidence-designer` 置信度设计通用skill；
- Excel 与 Word 两个设计示例。

暂不将统一运行时协议或统一置信度 SDK 作为首版范围。未来只有在建设中心化 Agent 平台、统一观测或统一审计时，再单独研究运行时接口标准。
