Internal Method Deck
01 · Why为什么做

Why

Agent 的瓶颈，正在从“生成”转向“验收”

没有结果置信度，业务只能在“全部相信”和“全部复核”之间二选一：前者风险不可控，后者抵消自动化收益。

Without confidence当前痛点
01
不知道哪些结果可以信

结果可靠程度混合，表达流畅也可能事实错误。

02
人工审查只能全量兜底

用户逐项翻输入、查来源、理解过程，验收成为新瓶颈。

03
“请人工检查”没有解决问题

不说明查哪个结果、对照什么、确认后如何处理。

The missing layer

结果级
置信度

不是给模型打分，而是给业务结果定使用与审查策略。

可以用需要查必须停
With confidence解决的问题
01
每个结果都有明确使用路径

区分直接使用、定点核查与确认阻断，不再一刀切。

02
人工只介入真正需要审查的地方

收缩审查范围，把验收时间用在不确定和高风险结果上。

03
审查对象、证据和动作都可定位

用户知道查什么、去哪里验证，并在错误进入下游前处理。

少审不再按最坏情况复核全部结果
审准审查聚焦已定位的结果、来源和不确定点
控风险为自动流转和人工介入建立明确边界
核心价值：置信度不是证明 Agent 永远正确，而是让业务在结果可靠程度混合的现实下，仍然能够安全、高效地使用 Agent。
生成只是起点 · 低成本验收才能真正进入业务
02 · Scope范围与边界

Scope

做什么，不做什么

做 本方法关注

  • 业务真正要求输出什么
  • 输出应拆成哪些结果单元
  • 每个结果如何追溯到实际来源
  • 每个结果在什么业务条件下属于高 / 中 / 低
  • 中、低结果需要用户检查什么

适用：信息提取、数据清洗计算、Excel/Word 生成、问卷摘要报告、分类判断结论建议等。

不做 明确排除

  • 整个文件或整个任务的总体置信度
  • 交付校验（能否打开、模板是否完整）
  • 输入 / 来源 / 中间步骤自身的置信度
  • 无业务校准的百分比分数
  • 模型对自身结果的主观自信
边界比技巧更重要 · 评错对象 = 方法失效
03 · Landing Path落地路径

Landing Path

落地路径：四个动作，而不是另建一套平台

把最佳实践嵌入现有 Skill、脚本、SQL、API、工作流和人工审核节点即可。

01

定义结果

明确业务输出与独立验收单位

02

建立溯源

查清实际链并定义必须保留的信息

03

业务校准

写出高、中、低事实条件与检查动作

04

嵌入判断

把规则放到最容易取得准确数据的环节

判断原则：公式由脚本复算，状态由权威 API 判断，语义一致性由模型结合可见证据判断，业务例外交给人工；模型不能通过评价自己的回答来认证正确。
不另建统一置信度平台 / 服务 · 嵌入现有环节即可落地
04 · Result Unit结果单元

Result Unit

结果单元 = 用户可独立验收的最小对象

定义

业务要求输出中，用户能够独立接受、退回或修正的最小业务对象。

可以是

  • 问卷中的一道答案
  • Word 字段 / 内容块
  • Excel 指标或记录
  • 分类、判断、结论、建议

默认不是

  • 输入与候选列表
  • 清洗中间表 / 临时 JSON
  • 匹配日志 / 处理步骤
  • 仅“可以检查”的对象

拆分原则

可分别验收 → 拆
用户能分别接受、退回或修正的内容，应分别定义

只能整体确认 → 不拆
不必拆到字段、句子或事实的无限细分

来源更多 ≠ 单元更多
步骤更多 ≠ 单元更多

一道答案引用三个历史来源，若用户仍整题确认，仍是一个结果单元。
中间产物转结果单元 · 必须同时是业务输出 + 可独立验收

Provenance Chain

溯源逻辑：来源 ──[形成关系]──▶ 结果单元

通用结构 · 只记录事实，不评级 最小 = 结果位置 + 实际来源位置 + 来源如何形成结果

业务结果

示例 · 数据计算和汇总

──[汇总计算]──▶
月度净销售额

计算过程中的标准化明细若不是业务要求独立交付和验收的输出，应作为结果溯源或实现数据，不为它们单独定义高 / 中 / 低。

来源可定位到

  • 文件 / 表 / 行 / 列 / 单元格
  • Word 章节、段落、字段
  • 库表、API 记录、已确认口径

形成关系示例

  • 原样提取 · 格式转换 · 映射
  • 公式计算 · 规则判断
  • 相似匹配 · 多源汇总 · 合并

红线

  • 无法建立溯源链 → 不能定为高
  • 不给来源 / 中间节点评级
  • 不把模型思维过程当证据
用户审查的始终是业务结果 · 溯源链帮助直接找到核对材料
06 · Levels高 / 中 / 低

Levels

统一语义，具体条件由业务定义

等级
通用语义
默认用户动作
高
满足该业务定义的直接使用条件
无需因置信度专门检查，或按策略少量抽查
中
未确认存在错误，但有明确不确定点或边界条件
只检查该结果及已定位的溯源信息
低
已发现错误，或命中不可接受的重大风险
人工确认、修正或阻断后续使用

条件必须可检查

只用用户或程序能核对的事实：实际来源、主体、年份、计算规则等。不写抽象分数。

「模型生成」不是等级

它只是形成方式。模型生成的结果仍可分别属于高、中或低。

高可信 ≠ 永不复核

关键结果仍可因业务影响或既有控制制度要求额外复核。

示例：问卷答案 · 高=来源一致可用 · 中=需确认年份/主体 · 低=冲突或错引
07 · Business Impact业务影响（可选）

Optional Dimension

置信度回答“要不要审”；业务影响回答“错了会怎样”

置信度

用户是否需要审查这个结果
高 / 中 / 低驱动默认审查动作

业务影响

如果结果错误，会造成什么后果
影响审查优先级与强度，不改变等级

何时使用

  • 不同结果出错后果明显不同
  • 业务需要区分优先 / 重点审查对象
  • 已有正式风险 / 严重度分类时可直接复用

不需要时可以完全不定义；优先写具体后果，避免再造一套易混淆的通用分级。

结果
置信度
影响与审查
合同付款金额
高
可能重大财务损失 → 仍按财务控制仔细复核
一般说明字段

局部文字修改 → 快速核对标记内容及来源

示例影响（非强制）

关键：财务 / 法律 / 合规 / 决策后果 → 即使高可信也可强制复核
重要：影响主要业务或下游动作 → 优先处理需审查项
一般：局部内容 → 按置信度常规审查

两个维度正交 · 影响不改等级，只改审查强度
08 · Design Flow推荐设计流程

Design Flow

八步：从业务背景到可执行判断

1
建立业务背景目标、使用者、输入输出、验收标准、支持范围、错误影响
2
列出候选结果单元含义、定位方式、用户确认单位、验收标准、可选影响
3
了解实际如何形成仅已有实现：结果写哪、用了哪些来源、发生了什么关系
4
确认最终结果单元校验是否真生成、能否独立验收、有无错拆/错合
5
明确必须保留的溯源来源精度、形成关系、查看路径；另列当前缺项
6
定义高 / 中 / 低条件直接写业务条件；缺事实要写清缺什么
7
说明中低怎么检查查哪个结果、看哪些来源、核什么、查完如何处理
8
指定判断环节SQL / 脚本 / API / 规则 / 人工节点——最易拿到准数据的环节
实施原则：大模型可辅助语义一致、冲突发现、来源是否充分；不能通过评价自己的回答来认证结果正确。设计与改代码是两个动作，未经授权不默认改运行时。
无实现时跳过第 3 步现状分析 · 不猜测、不代替业务设计流程

Scenarios

三种常见形态，同一套模型

01 · 供应商问卷

结果单元 = 单题答案

完整问卷只是容器。知识库、候选列表、匹配日志默认不是结果单元。

历史问答
──[相似匹配 / 主体年份调整]──▶

02 · 多文件填 Word

结果单元 = 字段 / 内容块

完整 Word 不评级。多文件共同形成时，溯源链关联全部实际来源。

源合同日期
──[提取 + 格式标准化]──▶
模板「合同结束日期」
03 · 数据汇总

结果单元 = 业务指标

标准化明细若非独立交付，进入溯源，不单独评级。

订单明细 + 退款记录 + 已确认计算口径
──[汇总计算]──▶
月度净销售额

场景不同，对象粒度不同 · 方法不变
10 · Review Actions中低如何检查

Review Actions

中、低必须写清「查什么、怎么处理」

检查说明四要素

  • 检查哪个结果（位置）
  • 查看哪些来源位置
  • 核对什么业务条件
  • 检查后如何处理

避免只写「请人工检查」。

可执行示例

  • 查问卷第 12 题，对照历史 Sheet2!A37:B37 的主体和年份
  • 核合同结束日期，查看源合同第 4 页第 8 条
  • 查净销售额，按已列订单、退款与口径复算
中定点核对
打开标记结果 → 对照已定位来源 → 确认边界条件是否成立 → 接受或降为低
低修正阻断
确认错误或风险 → 修正结果 / 更换来源 → 或阻断下游使用
高可直接用
默认不因置信度专门检查；关键项可按制度抽查或强制复核
运行时至少回答：哪个结果 · 什么等级 · 为何审 · 查什么 · 如何溯源
11 · Implementation开发与运行细节

Implementation

定义在开发期；判断在最易取数的现有环节

开发 / 维护阶段

业务置信度定义是规范

  • 结果单元、溯源要求、高中低条件、检查方式
  • 不要求运行流程动态读取定义文件
  • 不要求统一运行时协议 / 状态枚举 / 独立服务

运行阶段

由现有环节完成判断与展示

  • SQL · 生成脚本 · 业务代码 · 权威 API
  • 规则引擎 · 工作流节点 · 人工审核节点
  • 展示位置由业务定：批注 / 辅表 / 页面 / JSON / 日志

建议目录

.agent-confidence/
├── definition.md　← 用户已认可的当前定义
└── feedback.md　　← 真实问题 / 反馈 / 决定（只追加）

反馈演进

真实问题归档 → 分析影响的结果与条件 → 形成修改建议 → 用户确认 → 更新 definition

未确认建议必须与生效定义分离；不猜测问题，不凭空改规则。

判断放在最准、最近的数据环节 · 不为形式统一再建重复系统
12 · Anti-Patterns不推荐做法

Anti-Patterns

这些做法会让置信度失去业务意义

  • ×把输入、来源、候选、中间数据、处理步骤当作结果单元
  • ×因为“可以检查”就单独评级
  • ×给溯源链每个节点定义高中低
  • ×要求用户逐个检查溯源链所有节点
  • ×把模型思维过程或自评当作证据 / 等级
  • ×把「模型生成」与高中低并列
  • ×用百分比分数覆盖不同业务结果
  • ×给整文件 / 整任务算总体置信度或机械平均
  • ×只写「需要人工检查」，不写位置与动作
  • ×把未核实或尚未生成的内容写成当前事实
  • ×猜测测试问题；未确认就改生效定义
  • ×为形式统一强制同一套运行状态 / 接口 / 框架
一句话自检 · 评的是结果吗？条件是业务的吗？中低能定点查吗？
13 · Summary总结与清单

Takeaways

记住这五条，方法就不会跑偏

  1. 从业务输出出发，拆到用户可独立验收
  2. 每个结果可追溯；溯源链记录事实，不评级
  3. 只评结果：高中低由业务条件定义
  4. 中低必须可执行：位置 · 来源 · 条件 · 处理
  5. 判断放在最准环节；反馈真实且经确认才改定义

上线前快检（摘录）

  • 结果单元是否都来自业务明确输出？
  • 有无把中间步骤误当结果？
  • 是否误定义文件 / 任务总分？
  • 是否用模型自信替代业务条件？
  • 中低是否写清检查位置与动作？
  • 每条规则是否由最易取数环节判断？
把审查成本，压到真正需要审查的结果上。
