Internal Method Deck
01 · Why为什么做

Why

Agent 的瓶颈，正在从“生成”转向“验收”

没有结果置信度，业务只能在“全部相信”和“全部复核”之间二选一：前者风险不可控，后者抵消自动化收益。

Without confidence当前痛点
01
不知道哪些结果可以信

结果可靠程度混合，表达流畅也可能事实错误。

02
人工审查只能全量兜底

用户逐项翻输入、查来源、理解过程，验收成为新瓶颈。

03
“请人工检查”没有解决问题

不说明查哪个结果、对照什么、确认后如何处理。

The missing layer

结果级
置信度

不是给模型打分，而是给业务结果定使用与审查策略。

可以用需要查必须停
With confidence解决的问题
01
每个结果都有明确的处理方式

高置信度可直接使用，中置信度定点核查，低置信度人工确认或阻断下游。

02
人工只介入真正需要审查的地方

收缩审查范围，把验收时间用在不确定和高风险结果上。

03
需要检查时，明确告诉用户怎么查

指出要检查的结果、对应来源和处理动作，避免错误进入下游。

少审不再按最坏情况复核全部结果
审准审查聚焦已定位的结果、来源和不确定点
控风险为自动流转和人工介入建立明确边界
核心价值：置信度不是证明 Agent 永远正确，而是让业务在结果可靠程度混合的现实下，仍然能够安全、高效地使用 Agent。
生成只是起点 · 低成本验收才能真正进入业务
02 · Landing Path落地路径

Landing Path

落地路径：四个动作

把最佳实践嵌入现有 Skill、脚本、SQL、API、工作流和人工审核节点即可。

01

定义结果单元

明确业务输出与独立验收单位

02

建立溯源

查清实际链并定义必须保留的信息

03

业务校准

写出高、中、低事实条件与检查动作

04

嵌入Agent

把规则放到最容易取得准确数据的环节

不另建统一置信度平台 / 服务 · 嵌入现有环节即可落地
03 · Result Unit结果单元

Result Unit

结果单元——只评用户可独立验收的业务结果

结果单元，是业务要求输出中，用户能够独立接受、退回或修正的最小对象。

Business output 业务要求产出的结果

是用户最终需要看到或使用的内容。

+
Independent review 用户可独立验收

能够单独接受、退回或修正

=
Only rated object 结果单元

唯一需要定义高 / 中 / 低的对象

常见的结果单元

01问卷中的一道答案
02Word 字段或内容块
03Excel 指标或记录
04分类、判断、结论或建议

两个容易混淆的边界

×
整份文件或整个任务

它们通常只是多个结果的容器，不计算一个总置信度。

×
输入、来源和中间步骤

它们用于形成和验证结果，但不单独评级。

拆分原则

能分别验收 → 拆开分别定义结果单元

只能整体确认 → 不拆保持为一个结果单元

单元数量只看验收方式不看来源多少，也不看处理步骤多少

只有同时满足“业务输出 + 可独立验收”，才是结果单元
04 · Provenance结果溯源链

Provenance Chain

结果溯源——每个结果都要回答：来自哪里，如何形成

最小溯源 = 结果位置 + 实际来源位置 + 来源如何形成结果

只记录事实，不为链路节点评级
01 · Actual Source 实际来源

一个或多个，但必须可直接定位

02 · Formation 形成关系

写清从来源到结果发生了什么

03 · Result Unit 业务结果

有明确位置，用户可独立验收

唯一评级对象

LOCATE来源要定位到能直接打开的位置

文档类文件 → 章节 / 段落 / 字段

数据类表、库表或 API → 行 / 列 / 单元格 / 记录

EXPLAIN形成关系只写实际动作

直接处理原样提取 · 格式转换 · 映射

派生处理公式计算 · 规则判断 · 相似匹配 · 多源汇总

三条边界

链路不完整
不能定为高置信度

来源与中间节点
不单独定义高 / 中 / 低

模型思维过程
不能作为溯源证据

用户审查的始终是业务结果 · 溯源链帮助直接找到核对材料
05 · Levels高 / 中 / 低

Levels

统一语义，具体条件由业务定义

等级
通用语义
默认用户动作
高
满足该业务定义的直接使用条件
无需因置信度专门检查，或按策略少量抽查
中
未确认存在错误，但有明确不确定点或边界条件
只检查该结果及已定位的溯源信息
低
已发现错误，或命中不可接受的重大风险
人工确认、修正或阻断后续使用

条件必须可检查

只用用户或程序能核对的事实：实际来源、主体、年份、计算规则等。不写抽象分数。

「模型生成」不是等级

它只是形成方式。模型生成的结果仍可分别属于高、中或低。

高可信 ≠ 永不复核

关键结果仍可因业务影响或既有控制制度要求额外复核。

示例：问卷答案 · 高=来源一致可用 · 中=需确认年份/主体 · 低=冲突或错引
06 · Business Impact业务影响（可选）

Optional Dimension

置信度回答“要不要审”；业务影响回答“错了会怎样”

置信度

用户是否需要审查这个结果
高 / 中 / 低驱动默认审查动作

业务影响

如果结果错误，会造成什么后果
影响审查优先级与强度，不改变等级

何时使用

  • 不同结果出错后果明显不同
  • 业务需要区分优先 / 重点审查对象
  • 已有正式风险 / 严重度分类时可直接复用

不需要时可以完全不定义；优先写具体后果，避免再造一套易混淆的通用分级。

结果

示例影响（非强制）

关键：财务 / 法律 / 合规 / 决策后果 → 即使高可信也可强制复核
重要：影响主要业务或下游动作 → 优先处理需审查项
一般：局部内容 → 按置信度常规审查

两个维度正交 · 影响不改等级，只改审查强度
07 · Design Flow推荐设计流程

Design Flow

八步：从业务背景到可执行判断

1
建立业务背景目标、使用者、输入输出、验收标准、支持范围、错误影响
2
列出候选结果单元含义、定位方式、用户确认单位、验收标准、可选影响
3
了解实际如何形成仅已有实现：结果写哪、用了哪些来源、发生了什么关系
4
确认最终结果单元校验是否真生成、能否独立验收、有无错拆/错合
5
明确必须保留的溯源来源精度、形成关系、查看路径；另列当前缺项
6
定义高 / 中 / 低条件直接写业务条件；缺事实要写清缺什么
7
说明中低怎么检查查哪个结果、看哪些来源、核什么、查完如何处理
8
指定判断环节SQL / 脚本 / API / 规则 / 人工节点——最易拿到准数据的环节
实施原则：大模型可辅助语义一致、冲突发现、来源是否充分；不能通过评价自己的回答来认证结果正确。设计与改代码是两个动作，未经授权不默认改运行时。
无实现时跳过第 3 步现状分析 · 不猜测、不代替业务设计流程
08 · Scenarios典型场景

Scenarios

三种常见形态，同一套模型

01 · 供应商问卷

结果单元 = 单题答案

完整问卷只是容器。知识库、候选列表、匹配日志默认不是结果单元。

历史问答
──[相似匹配 / 主体年份调整]──▶
当前单题答案
02 · 多文件填 Word

结果单元 = 字段 / 内容块

完整 Word 不评级。多文件共同形成时，溯源链关联全部实际来源。

源合同日期
──[提取 + 格式标准化]──▶
模板「合同结束日期」
03 · 数据汇总

结果单元 = 业务指标

标准化明细若非独立交付，进入溯源，不单独评级。

订单明细 + 退款记录 + 已确认计算口径
──[汇总计算]──▶
月度净销售额
共同模式：业务输出 → 用户确认单位 → 结果单元 → 实际来源与形成关系 → 业务条件定义高中低 → 中低给出可执行检查动作。
场景不同，对象粒度不同 · 方法不变
09 · Review Actions中低如何检查

Review Actions

中、低必须写清「查什么、怎么处理」

检查说明四要素

  • 检查哪个结果（位置）
  • 查看哪些来源位置
  • 核对什么业务条件
  • 检查后如何处理

避免只写「请人工检查」。

可执行示例

  • 查问卷第 12 题，对照历史 Sheet2!A37:B37 的主体和年份
  • 核合同结束日期，查看源合同第 4 页第 8 条
  • 查净销售额，按已列订单、退款与口径复算
中定点核对
打开标记结果 → 对照已定位来源 → 确认边界条件是否成立 → 接受或降为低
低修正阻断
确认错误或风险 → 修正结果 / 更换来源 → 或阻断下游使用
高可直接用
默认不因置信度专门检查；关键项可按制度抽查或强制复核
运行时至少回答：哪个结果 · 什么等级 · 为何审 · 查什么 · 如何溯源
10 · Feedback Loop测试问题与用户反馈

Feedback Governance

真实问题驱动规则演进，用户确认后才生效

开发测试、用户自测和验收反馈可以完善置信度定义；但不猜测问题，也不让未确认建议直接改变当前规则。

01

提供真实问题

来自开发测试、用户自测或验收反馈

02

保留原始描述

先完整归档，不用分析结论替换用户原话

03

分析影响范围

定位可能影响的结果单元和置信度条件

04

形成修改建议

建议转化为高中低条件或具体检查方式

05

用户确认

由业务用户决定采用、调整或拒绝建议

06

更新生效定义

只将已确认的规则写入当前业务定义

两份文件，分开管理

.agent-confidence/
├── definition.md　← 已确认的当前定义与溯源要求
└── feedback.md　← 原始问题、分析、决定与最终结果（只追加）

三条治理边界

  • 没有真实记录，不凭空建立问题档案
  • 未确认建议与 definition.md 严格分离
  • 现状分析、实现缺项和单次运行溯源不写入这两份业务级文件
真实记录 · 建议与生效定义分离 · 用户确认后才更新
11 · Anti-Patterns不推荐做法

Anti-Patterns

这些做法会让置信度失去业务意义

  • ×把输入、来源、候选、中间数据、处理步骤当作结果单元
  • ×因为“可以检查”就单独评级
  • ×给溯源链每个节点定义高中低
  • ×要求用户逐个检查溯源链所有节点
  • ×把模型思维过程或自评当作证据 / 等级
  • ×把「模型生成」与高中低并列
  • ×用百分比分数覆盖不同业务结果
  • ×给整文件 / 整任务算总体置信度或机械平均
  • ×只写「需要人工检查」，不写位置与动作
  • ×把未核实或尚未生成的内容写成当前事实
  • ×猜测测试问题；未确认就改生效定义
  • ×为形式统一强制同一套运行状态 / 接口 / 框架
一句话自检 · 评的是结果吗？条件是业务的吗？中低能定点查吗？
