# 结果级置信度方法

## 核心流程

```text
业务需求
→ 明确业务输出
→ 先列出可能的结果单元
→ （已有业务实现时）分析当前实际结果溯源链
→ 确认最终的结果单元
→ 明确每个结果必须保留哪些溯源信息
→ （已有业务实现时）检查当前实现还缺什么
→ 定义高、中、低和具体检查方式
→ 由最容易取得准确数据的现有环节完成判断
```

业务 Skill、脚本或工作流已经实现时，必须先查清当前实际生成了什么结果、实际来源在哪里、来源怎样形成结果，以及结果写在哪里。使用这些事实确认结果单元后，再设计高、中、低。

没有已实现业务流程时，跳过当前实际链分析和当前实现检查。不要猜测当前链，也不要代替业务 Skill 设计处理流程。

完整文件只用于汇集和呈现多个结果，不定义总体置信度。

## 结果单元

先从业务输出中列出可能的结果单元。最终结果单元必须同时满足：

- 属于业务明确要求的输出；
- 用户能够独立接受、退回或修正；
- 具有明确结果位置和业务验收标准。

已有业务实现时，还要使用当前实际结果溯源链检查候选项是否真的由当前流程生成。仅留空、没有实际来源或仍待人工完成的内容，不写成当前已经生成并可评级的结果。

输入、来源、候选集、中间数据、日志和处理步骤默认不是结果单元。只有它们本身也是业务要求独立使用和验收的输出时，才转为结果单元。

## 结果溯源链

每个结果单元都必须能够记录和展示：

```text
结果位置
+ 至少一个实际来源位置
+ 来源怎样形成结果
+ 用户怎样查看这些信息
```

已有业务实现时，在确认结果单元前按结果类型和实际形成方式了解当前实际链。只记录现有材料能够证明的实际来源，候选来源不算实际来源，无法确认的内容写“未知”。这一步不要求在设计文档中穷举每个运行结果。

一个来源可以形成一个或多个结果，一个结果也可以使用一个或多个来源，不要求固定的线性步骤。结果溯源链是事实记录，不定义高、中、低，也不包含模型内部思维过程。

设计中应区分当前实际链和每次运行必须保留的溯源信息；已有实现时再简要指出当前缺少什么。`.agent-confidence/definition.md` 只保存用户确认的长期要求，某次运行的具体来源和结果位置保存在运行记录中。

## 高、中、低

- **高**：满足业务直接使用条件；
- **中**：未确认存在错误，但有明确不确定点或边界条件，需要检查指定结果及其来源；
- **低**：有明确错误或不可接受风险，需要确认、修正或阻断。

高可信条件必须要求该结果能够定位到实际来源及来源怎样形成结果。无法建立结果溯源链的结果不能判为高可信。

等级只属于业务结果。具体条件由业务验收标准决定，并且只能使用用户或程序能够检查的事实，例如实际来源、适用主体、数据年份和计算规则。不要使用模型主观自信、无校准百分比或“是否由模型生成”代替业务条件。

## 可选业务影响

业务影响用于描述结果出错的具体后果。仅在业务需要区分重点结果时使用，不统一规定等级，也不改变结果置信度。

## 真实反馈

测试问题和用户反馈必须来自真实记录，不能凭空假设。收到真实问题后先归档并提出修改建议；只有用户确认后，才更新当前定义。

## 实施原则

- 优先复用现有 SQL、脚本、代码、API、规则和人工流程；
- 每条置信度条件由最容易取得所需准确数据的现有环节判断；
- 大模型只辅助适合文字含义判断的条件，不能通过评价自己的回答来证明结果正确；
- 不建立文件或任务总体等级；
- 不要求统一运行时协议或独立置信度服务；
- 不默认修改实际代码。
