# 结果级置信度方法

## 核心模型

```text
业务需求
→ 明确业务输出
→ 识别用户可独立接受、退回或修正的结果单元
→ 为每个结果定义结果溯源要求
→ 为每类结果定义高、中、低和定向审查动作
→ 将判断逻辑放到最接近业务事实的位置
```

完整文件是结果的承载容器，不定义总体置信度。

## 结果单元

结果单元必须同时满足：

- 属于业务明确要求的输出；
- 用户能够独立接受、退回或修正；
- 具有明确结果位置和业务验收标准。

输入、来源、候选集、中间数据、日志和处理步骤默认不是结果单元。只有它们本身也是业务要求独立使用和验收的输出时，才转为结果单元。

## 结果溯源链

每个结果单元必须记录：

```text
结果位置
+ 至少一个来源位置
+ 来源—结果形成关系
```

结果溯源链可以是一对一、一对多或多对一关系，不要求固定步骤。它是事实记录，不定义高、中、低，也不包含模型内部思维过程。

## 高、中、低

- **高**：满足业务直接使用条件；
- **中**：有明确不确定点，需要定向审查；
- **低**：有明确错误或不可接受风险，需要确认、修正或阻断。

等级只属于业务结果。具体条件由业务验收标准决定，不使用模型主观自信、无校准百分比或“是否由模型生成”代替。

## 可选业务影响

业务影响用于描述结果出错的具体后果。仅在业务需要区分重点结果时使用，不统一规定等级，也不改变结果置信度。

## 真实反馈

不要猜测测试问题或用户反馈。只有收到真实问题时才归档并提出修改建议；用户确认后，才把建议沉淀到当前定义。

## 实施原则

- 优先复用现有 SQL、脚本、代码、API、规则和人工流程；
- 将判断逻辑放到最容易取得准确事实的位置；
- 大模型只辅助适合语义判断的条件，不进行自我认证；
- 不建立文件或任务总体等级；
- 不要求统一运行时协议或独立置信度服务；
- 不默认修改实际代码。
