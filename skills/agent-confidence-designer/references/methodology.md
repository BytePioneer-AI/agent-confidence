# 可信等级设计方法

## 1. 设计目标

把业务人员的全文逐项检查转成按具体结果和风险位置定向检查。通用 Skill 只负责设计，不规定运行时技术框架。

## 2. 推荐分析顺序

1. 业务上下文与阶段；
2. 结果单元与结果形态；
3. 可信含义和限制；
4. 历史问题归因与可泛化性；
5. 判断维度和证据；
6. 硬性规则；
7. 高 / 中 / 低；
8. 业务影响和审查动作；
9. 现有系统实施位置；
10. 真实终版或下游结果验证。

## 3. 适用性优先

很多结果“类型正确”但“当前上下文不适用”。设计时重点检查主体、时间、客户、用途、规则范围和数据来源。

## 4. 集合型结果

集合型结果必须定义期望数量与必要覆盖维度。数量达标不代表覆盖充分，数量不足也不一定错误，需结合业务矩阵判断。

## 5. 硬性规则与加权分

加权分只能作为辅助。主体不一致、关键规则遗漏、高影响事实无权威来源等条件应直接限制等级或自动处理动作。

## 6. 离线校准

有 AI 初稿与人工终版时，可用原样采用、格式调整、实质修改、人工补充、删除和未解决等标签评估等级规则。未校准前不对外展示概率。
