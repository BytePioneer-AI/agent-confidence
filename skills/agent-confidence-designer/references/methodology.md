# 方法论

## 目标

指导开发人员在具体业务中定义高、中、低可信等级，并将判断逻辑融入现有系统。

## 推荐顺序

```text
业务背景
→ 结果拆分
→ 关键性
→ 测试问题
→ 风险条件
→ 各结果高/中/低
→ 整体组合原则
→ 实现位置
→ 人工审查动作
```

## 通用层与业务层

通用层负责：

- 分析流程；
- 风险分类；
- 输出结构；
- 设计审查。

业务层负责：

- 什么是正确；
- 哪些结果关键；
- 哪些条件对应高、中、低；
- 判断逻辑如何在现有 SQL、代码、API、规则或工作流中实现；
- 什么情况下阻断或人工确认。

## 三个等级

- **高**：符合该业务定义的直接使用条件；
- **中**：存在需要定向核查的边界或风险；
- **低**：命中明确错误或严重业务风险。

不要使用没有校准依据的百分比，也不要假设所有业务共享同一套技术协议。

## 结果拆分

结果粒度应满足：

- 能独立验收；
- 能定位风险；
- 能给出具体检查动作；
- 不会细到用户无法理解。

## 历史问题进入设计

每个问题至少转化为：

```text
触发条件
+ 影响结果
+ 严重度
+ 可识别方式
+ 等级影响
+ 审查动作
+ 回归测试建议
```

## 实施原则

- 优先复用现有系统；
- 判断逻辑靠近业务事实和权威数据；
- 不为追求形式统一而重复建设；
- 只有在平台化需求明确时，才考虑统一运行时协议；
- 大模型可以辅助语义分析，但不以模型主观自信代替业务定义。
