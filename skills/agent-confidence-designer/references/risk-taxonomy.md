# 风险分类参考

使用风险分类帮助整理测试问题，但不要让分类替代具体业务描述。

## 输入风险

- 文件或记录遗漏；
- Schema、格式或编码异常；
- 字段含义不明确；
- 输入超出支持范围；
- 数据源过期或版本错误。

## 处理风险

- 字段映射错误；
- 日期、币种或单位转换错误；
- 错误去重、过滤或合并；
- 处理步骤遗漏；
- 工具或 API 返回异常内容。

## 计算风险

- 公式范围错误；
- 聚合口径错误；
- 明细与汇总不一致；
- 时间窗口错误；
- 舍入、精度或币种处理错误。

## 来源与一致性风险

- 输出无法追溯到输入；
- Word、Excel、数据库或 API 之间不一致；
- 使用了旧版本数据；
- 多个来源口径冲突。

## 内容与语义风险

- 事实描述与数据矛盾；
- 将相关性写成因果性；
- 结论缺少支持；
- 摘要遗漏关键限制；
- 建议建立在错误事实上。

## 输出与交付风险

- 文件无法打开；
- 关键章节、工作表或字段缺失；
- 格式导致下游无法使用；
- 输出未满足用户明确要求。

## 严重度参考

- `critical`：可能造成重大财务、法律、合规或关键业务后果；
- `high`：影响主要结论或关键下游动作；
- `medium`：需要修正，但一般不会直接造成重大后果；
- `low`：主要影响格式、表达或体验。

## 可识别方式参考

问题可以通过一种或多种方式识别：

- SQL；
- 现有业务代码；
- 数据库约束；
- API 或权威系统；
- 文件或文档处理；
- 日志和任务元数据；
- 规则引擎；
- 大模型语义辅助；
- 人工审核。

不要要求所有方式转换成统一验证器接口。
