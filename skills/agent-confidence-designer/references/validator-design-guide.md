# 验证器设计指南

## 1. 验证器类型

### `deterministic`

用于可复算、可比对、可通过业务规则明确判断的内容。它是高可信的主要证据。

### `semantic_assist`

用于结论依据、语义矛盾、摘要忠实度等固定脚本难以覆盖的问题。必须输出结构化风险和证据，不能直接决定最终等级。

### `manual_gate`

用于高风险且暂时无法自动验证的内容，或业务制度要求人工批准的内容。

## 2. 每个验证器必须定义

- 唯一 ID 和名称；
- 类型与实现状态；
- 适用结果单元；
- 输入和依赖；
- 检查逻辑；
- PASS 条件；
- FAIL 时等级影响；
- 不可用时等级影响；
- 证据字段；
- 代码位置；
- 回归测试。

## 3. 状态协议

统一返回：

- `PASS`
- `FAIL`
- `NOT_RUN`
- `UNSUPPORTED`
- `ERROR`
- `NOT_APPLICABLE`

验证器必须显式区分“没有发现问题”和“没有执行检查”。

## 4. 设计检查

### 对 critical 结果

- 至少配置一个必要的确定性验证器；
- 语义检查不能成为唯一证据；
- 任一必要验证失败应判低；
- 必要验证不可用时不能判高；
- 必须保留来源和定位证据。

### 对语义验证器

要求输出：

```json
{
  "status": "RISK",
  "target": "经营分析第3段",
  "risk_type": "unsupported_causal_claim",
  "reason": "数据只显示销售下降，没有客户需求数据支持原因判断",
  "evidence": ["销售汇总表!D18"],
  "review_action": "核对客户订单量或改为不确定性表述"
}
```

### 对人工门禁

定义：

- 谁审批；
- 审批什么；
- 使用哪些证据；
- 通过和拒绝条件；
- 审批记录如何关联结果。

## 5. 验证器反模式

- 只检查脚本是否运行成功；
- 只检查文件是否存在；
- 使用执行 Agent 的自评作为唯一依据；
- PASS 条件含糊；
- 无法定位错误对象；
- 忽略验证器自身异常；
- 没有错误样本和边界样本测试；
- 风险命中后只输出“请检查”。
